2/**
3 * MediaAdapter — normalise non-image attachments into pipeline-compatible shapes.
4 *
5 * The Pipeline was originally designed for image/jpeg and image/png attachments.
6 * This adapter sits in front of the pipeline loop and converts other media types
7 * into a form the pipeline can process:
8 *
9 *   image/*          → passthrough with media_type = 'image' (no conversion needed).
10 *   application/pdf  → rasterise each page with Imagick, emit one 'image' entry per page.
11 *   video/*          → extract the first frame with ffmpeg as a poster/description
12 *                       image, but keep the original video file — emit one 'video'
13 *                       entry carrying both. The video itself is what gets published;
14 *                       the frame is only used for AI description and the <video poster>.
15 *   audio/*          → passthrough with media_type = 'audio' (pipeline routes to audio branch).
16 *
17 * Attachments that cannot be converted (Imagick unavailable for PDFs) are logged and
18 * dropped — the caller receives fewer entries than it passed in. Video is more
19 * forgiving: if ffmpeg is unavailable or frame extraction fails, the video itself is
20 * still forwarded (with no poster frame) rather than being dropped entirely — Pipeline
21 * falls back to a text-only description in that case, same as it does for audio with
22 * no transcript available.
23 *
24 * Each attachment array is expected to have at minimum:
25 *   'data'       string  Raw binary content.
26 *   'mime'       string  MIME type (matches Parser output).
27 *   'filename'   string  Original filename.
28 *
29 * Each returned entry carries those same keys plus:
30 *   'media_type'  string  'image' | 'audio' | 'video' — tells Pipeline which branch to use.
31 *   'mime'        string  Possibly updated MIME (e.g. 'image/jpeg' after PDF rasterisation).
32 *   'poster_data' string  (video only) Extracted first-frame JPEG binary, or '' if unavailable.
33 *   'poster_mime' string  (video only) Always 'image/jpeg' when poster_data is non-empty.
34 *
35 * Also home to maybe_downscale_for_vision() — a per-request image-downscale
36 * helper for vision API calls (cost optimisation), deliberately NOT wired
37 * into adapt()/adapt_pdf()/adapt_video() themselves. See that method's own
38 * doc for why: the attachment 'data'/'poster_data' this class returns is
39 * reused downstream for enhancement and for the actual published file, so
40 * mutating it here would silently shrink published artwork.
41 *
42 * @package Agnosis\AI
43 */

95     * Downscale an image to the admin-configured max width (agnosis_ai_vision_max_width_px,
96     * default 800px) for a single vision-API request — a pure cost/latency
97     * optimisation, since vision-token cost scales with resolution and a
98     * description task needs nowhere near a photo's original resolution.
99     *

101     * implementation, immediately before base64-encoding the request body,
102     * and use only the return value for that one request. Never call it here
103     * in MediaAdapter (i.e. never let its output replace an attachment's
104     * 'data'/'poster_data') and never let its output flow back into
105     * Pipeline::process_single()'s $image_data / process_video_single()'s
106     * $poster_data. Those same variables are also used, unmodified, for:
107     * the image enhancement API call (which needs real resolution to do a
108     * good job), Pipeline's 'original_data' result (the actual file
109     * PostCreator publishes when enhancement doesn't run), and — for video —
110     * the poster frame's own 'poster_data' result (the real `<video poster>`
111     * attachment). Downscaling any of those in place would silently publish
112     * artwork/poster images at only agnosis_ai_vision_max_width_px wide,
113     * which is not what this setting is for.
114     *
115     * Deliberately conservative: never upscales (a source already narrower
116     * than the configured max is returned untouched), and any failure —
117     * disabled via the '0' setting, Imagick unavailable, or a corrupt/
118     * unreadable blob — falls back to returning the original image
119     * unmodified rather than ever blocking a describe() call. Mirrors
120     * adapt_pdf()'s existing Imagick-availability guard and logging
121     * convention, since Imagick is already a soft dependency in this class.
122     *
123     * @param string $data Raw image binary.
124     * @param string $mime Image MIME type (informational only — Imagick
125     *                     preserves the source format on resize).
126     * @return string Resized image binary, or the original $data unchanged
127     *                if downscaling is disabled, unnecessary, or unavailable.
128     */
129    public static function maybe_downscale_for_vision( string $data, string $mime ): string {

133            return $data; // Disabled via the admin setting, or nothing to resize.
134        }
135

137            // No warning logged here (unlike adapt_pdf()'s hard failure) —
138            // sending the original, undownscaled image is a silent, harmless
139            // fallback, not a broken feature; PDF rasterisation without
140            // Imagick has no such fallback, which is why that path warns.
141            return $data;

258     * stdin/stdout, for reliable cross-format frame extraction.
