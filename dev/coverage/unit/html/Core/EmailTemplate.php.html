1<?php
2/**
3 * Shared branded HTML email shell.
4 *
5 * Every HTML email the plugin sends (Publishing\Notification,
6 * Artist\AdmissionNotification, Artist\DepartureNotification,
7 * Artist\CommunityCapNotification, Artist\CommunityBroadcast, and both
8 * Newsletter\Mailer digests) previously hand-rolled the same 600px branded
9 * card — DOCTYPE, meta tags, header/body/footer table rows, the
10 * `#0d0d12`/`#7c6af7` color pair, and the CTA button style string —
11 * independently in each `build_*_body()` method. That worked but meant (a)
12 * the same ~20 lines of boilerplate were duplicated a dozen-plus times, and
13 * (b) the header/accent colors were literal string constants, so an
14 * operator who wanted to re-brand outgoing mail (a common ask once a site
15 * has its own visual identity) had no lever anywhere — changing the accent
16 * meant editing PHP in a dozen files by hand.
17 *
18 * This class is that shared shell now — genuinely, for every class listed
19 * above: `render()` builds the full HTML document around a caller-supplied
20 * body (and optional extra header/footer content), reading every visual
21 * value it renders from Settings → Branding instead of a hardcoded literal:
22 * header background, accent color, card width, page/card/footer background,
23 * primary/secondary text color+size, divider/border color, button text
24 * color, and notice/info-box background (`agnosis_email_header_bg`,
25 * `agnosis_email_accent`, `agnosis_email_width`, `agnosis_email_body_bg`,
26 * `agnosis_email_card_bg`, `agnosis_email_footer_bg`,
27 * `agnosis_email_text_color`, `agnosis_email_text_size`,
28 * `agnosis_email_text_secondary_color`, `agnosis_email_text_secondary_size`,
29 * `agnosis_email_border_color`, `agnosis_email_button_text_color`,
30 * `agnosis_email_notice_bg`, `agnosis_email_footer_label_color`), so every
31 * email re-brands together the moment any setting changes — not just the
32 * shell's own header/body/footer rows, but the decorative notice/quote boxes
33 * and dividers every `build_*_body()` method dots throughout its content,
34 * which used to carry their own independent `#f9f9f9`/`#eee` literals no
35 * setting ever reached. `card()` is
36 * the same markup minus the DOCTYPE/html/head/body wrapper, as a reusable
37 * fragment — `Newsletter\Mailer::build_body()` calls it directly since
38 * `Newsletter\Archive`'s "view this issue online" page needs the identical
39 * card without a second document wrapper. `button()` is the matching shared
40 * CTA-button renderer, replacing each file's own repeated `$btn_base`
41 * style-string pattern. `Core\EmailFooter` (the "work emails" reference card
42 * several of the classes above append as `$footer_extra_html`) also reads
43 * `accent()`, `footer_label_color()`, and `text_secondary_color()` for its
44 * mailto link, bold labels, and description/preferences-link colors, so a
45 * rebranded install doesn't end up with the new accent on every button but
46 * the old purple/greys still showing in the footer underneath it.
47 *
48 * `EmailBranding::header_subtitle_color()` (a small public method on the
49 * sibling class, not this one) covers one thing Settings → Branding
50 * deliberately does NOT expose a raw field for: the muted subtitle line
51 * under the header wordmark (`Newsletter\Mailer`'s "Community Newsletter"
52 * heading, `Artist\Invitation`'s "You're invited" line). That color has to
53 * track `header_bg()`'s own contrast, the same way the wordmark fallback
54 * itself does — a fixed literal or an independently-configurable field would
55 * both risk the exact invisible-text failure AUDIT-0.9.29.md §2b fixed for
56 * the wordmark, just for the subtitle instead.
57 *
58 * Deliberately NOT touching the destructive/"danger" red (`self::DANGER`)
59 * with a setting — unlike the header/accent pair, that color is a
60 * semantic signal (an irreversible action: reject, remove, vote-to-remove)
61 * rather than a brand choice, and every email in the plugin already uses
62 * the exact same red for exactly that meaning. Making it configurable
63 * would invite an operator to make destructive actions LESS visually
64 * distinct, which is the opposite of what it's for. Also deliberately fixed:
65 * the white backing panel behind an uploaded logo (`EmailBranding`) — a
66 * functional contrast device against the (possibly dark) header bar, not a
67 * brand choice, so it stays white regardless of card_bg().
68 *
69 * @package Agnosis\Core
70 */

268     * owns the shell, not the per-email content.
