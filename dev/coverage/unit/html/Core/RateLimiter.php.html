2/**
3 * Transient-based per-IP rate limiter.
4 *
5 * Counts requests per action per IP address within a window and returns a
6 * WP_Error when the limit is exceeded.  Uses WordPress transients so it works
7 * on any host without extra infrastructure.
8 *
9 * This is a FIXED window, not a sliding one (code-quality audit §6 —
10 * corrected from an earlier docblock that called it "sliding"/"rolling"):
11 * transient_key()'s `floor( time() / $window_seconds )` slot changes the
12 * whole counter's identity once per window boundary, rather than continuously
13 * decaying older requests out of a moving lookback period. The practical
14 * effect is that a burst straddling a boundary (e.g. the limit's worth of
15 * requests just before a slot rolls over, plus the limit's worth again just
16 * after) can let up to roughly 2x the configured limit through inside one
17 * real-world window-length span. Acceptable at this plugin's actual traffic
18 * (a handful of low-single-digit-per-minute application/webhook limits, not a
19 * high-value target where a 2x burst materially changes the threat model) —
20 * documented here rather than switched to a heavier sliding-window
21 * implementation.
22 *
23 * Usage in a permission callback:
24 *
25 *   $rate = RateLimiter::check( 'admission_apply', 5, 60 );
26 *   if ( is_wp_error( $rate ) ) {
27 *       return $rate;
28 *   }
29 *
30 * IP detection uses REMOTE_ADDR only — not X-Forwarded-For — to prevent
31 * spoofing.  Sites behind a trusted reverse proxy that rewrites REMOTE_ADDR
32 * (e.g. via nginx real_ip_from) work correctly without any additional config.
33 *
34 * @package Agnosis\Core
35 */
