2/**
3 * Daily vote-email digest for artists in "digest" mode (security audit
4 * §5b/§4a).
5 *
6 * AdmissionNotification::on_application_received() sends one vote email per
7 * artist the instant an application is confirmed by the community — on an
8 * active node (daily applications) this is exactly the kind of volume §5b
9 * flags: an artist with no way to reduce it short of a full departure has no
10 * realistic exit but the spam button, the single worst outcome for the
11 * shared domain's reputation. NotificationPreferences lets an artist switch
12 * to `_agnosis_vote_email_mode = 'digest'`; this class is what that mode
13 * actually delivers — one email per day per digest-mode artist, listing
14 * every still-open application they haven't voted on yet, instead of one
15 * email per application as it arrives.
16 *
17 * Deliberately NOT built as a queue/delta ("what's new since their last
18 * digest") — every open application a digest-mode artist hasn't voted on is
19 * re-derived fresh on every run instead:
20 *   SELECT ... WHERE status = 'pending' AND NOT EXISTS (a vouch by this artist)
21 * This is simpler and self-healing (an artist who switches into digest mode
22 * mid-window, or whose earlier digest send failed, still sees every
23 * outstanding application on the very next run — nothing can be silently
24 * dropped) at the cost of one small, accepted edge case: an artist who was in
25 * 'instant' mode when an application arrived (and so already got that one
26 * individually) but switches to 'digest' before voting will see it once more
27 * in their next digest. That's a harmless repeat reminder, not a duplicate
28 * side effect — voting itself is idempotent (Admission::record_vote() updates
29 * the same row) — so it isn't worth a tracking table to prevent.
30 *
31 * @package Agnosis\Artist
32 */
