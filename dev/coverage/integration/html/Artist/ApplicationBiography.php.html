22 * An applicant already writes a short bio, an artist statement, and
23 * (optionally) a portfolio URL on the application form — see
24 * Admission::apply(). Today none of that ever becomes visible to a site
25 * visitor unless the artist separately emails bio@ after being admitted;
26 * it just sits in agnosis_applications. This class surfaces it: the moment
27 * an application is admitted (either path — Admission::admin_admit() or the
28 * community vouch-threshold path, Admission::maybe_admit() — both fire the
29 * same 'agnosis_artist_admitted' action), it builds a first agnosis_biography
30 * draft from that data, so a newly admitted artist starts with something
31 * rather than a blank profile.
32 *
33 * The draft goes through the exact same review pipeline every other Agnosis
34 * post uses: a '_agnosis_review_token'/'_agnosis_review_expiry' pair and the
35 * 'agnosis_post_drafted' action, picked up by Publishing\Notification, which
36 * emails the artist an Approve & Publish / Edit before publishing / Discard
37 * link (Publishing\ReviewEndpoints). Nothing here bypasses that review step —
38 * the artist always gets to see and approve (or edit, or discard) this
39 * biography before it goes live, same as an AI-drafted artwork.
40 *
41 * The portfolio URL becomes a wp:embed block at the end of the post, gated by
42 * the same Publishing\EmbedPolicy used for artist-submitted links in artwork/
43 * biography/event emails: embedded immediately if the host is on the site's
44 * trusted-platform list, otherwise only if the admin has enabled AI review
45 * and the AI approves it against the site's configured disallowed
46 * categories. A portfolio link is no more automatically trustworthy than any
47 * other artist-submitted link — the same policy, the same settings, applies
