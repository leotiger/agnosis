199    /**
200     * Whether a message looks like a reply or a forward carrying quoted
201     * content, rather than an original, curated submission (2026-07-15;
202     * forward detection widened 2026-07-14 — eighth audit §3b).
203     *
204     * Agnosis's own outbound mail (submission-review confirmations etc.)
205     * deliberately sets Reply-To back to the artist's own intake address
206     * (`Core\CommunityMailer::reply_to_header_for_post()`), specifically so
207     * hitting "reply" starts a new submission — but a mail client's default
208     * reply behaviour quotes the entire original message and prefixes the
209     * subject with "Re:", which is never what we want published verbatim:
210     * the quoted portion is not original content, and is often our own
211     * previous email verbatim. The policy (both this method's own name and
212     * the 0.9.25 changelog) has always been framed as "reply **or
213     * forward**", but until eighth-audit §3b nothing actually matched a
214     * forward as such — only a reply's own signals happened to also catch
215     * some forwards incidentally (e.g. Outlook reuses its reply separator
216     * for forwards too). Five independent signals, any one of which is a
217     * match:
218     *
219     * A match here is no longer an automatic reject on its own (eighth
220     * audit §3c, 2026-07-14) — the caller (`Email\Parser`) now runs a match
221     * through `extract_original_content()` below first, which pulls out
222     * whatever the sender actually wrote above the quoted/forwarded
223     * portion; only a message with NOTHING left after extraction (no
224     * original text, no attachment) is actually rejected. This method's own
225     * job stays exactly what its name says — detect — the accept/extract/
226     * reject decision lives one level up.
227     *
228     *   1. Subject starts with "Re:" or "Fwd:"/"Fw:" (any capitalisation,
229     *      optional space before the colon) — the standard reply/forward
230     *      prefix convention nearly every mail client adds.
231     *   2. Subject contains "[Agnosis]" anywhere — catches our own outbound
232     *      notification subjects (several of which are literally prefixed
233     *      this way) being replied to or forwarded verbatim without editing
234     *      the subject. Deliberately a DIFFERENT literal string than the
235     *      "[Biography]"/"[Event]"/"[Photo]"/"[Pure]" subject-line
236     *      indicators (PostCreator::INDICATORS) — no collision with that
237     *      unrelated, intentional feature.
238     *   3. Body contains a quoted-reply attribution line — either the
239     *      "On <date>, <name> <<email>> wrote:" style nearly every mail
240     *      client (Apple Mail, Gmail, etc.) inserts above quoted text, or
241     *      Outlook's "-----Original Message-----" separator (Outlook reuses
242     *      this same separator for a forward, so Outlook forwards were
243     *      already covered before this widening).
244     *   4. Body contains Gmail's own forward separator,
245     *      "---------- Forwarded message ---------" (hyphen count varies
246     *      slightly by client version, hence `-{2,}` rather than a fixed
247     *      count) — previously the one gap the policy's own docblock and
248     *      changelog claimed was covered but wasn't: a Gmail forward with no
249     *      "On … wrote:" line (i.e. the forwarder added no comment of their
250     *      own above the separator) sailed straight through.
251     *   5. Body contains Apple Mail/iOS Mail's own forward marker, "Begin
252     *      forwarded message:" — the same class of gap as #4, for the other
253     *      mail client the "On … wrote:" pattern's own docblock already
254     *      names as a reference implementation.
255     *
256     * False-positive risk is much smaller than it was before §3c: an artist
257     * who genuinely titles a piece "Re: Something" no longer loses the
258     * submission outright — extraction still finds their real body/subject
259     * content and processes it (their literal "Re: Something" title does
260     * get its "Re:" prefix stripped, the one remaining, deliberately
261     * accepted quirk of this policy). Only a message that resolves to
262     * NOTHING after extraction is rejected; that rejection email (see
263     * Publishing\Notification::on_submission_looks_like_reply()) tells the
264     * sender plainly to resend as a fresh, original message rather than a
265     * reply/forward, so even that remaining case is recoverable, not a
266     * silent loss.
267     *
268     * @param string $subject Message subject.
269     * @param string $body    Plain-text body.

304     * §3c, option (c) — the resolution chosen for the product tension
305     * between the 0.9.22 Reply-To "just hit reply to submit again" feature
306     * and the 0.9.25 blanket reply/forward rejection).
307     *
308     * Rather than outright rejecting every message is_reply_or_quote()

310     * comment above a reply's quoted thread, or above a forward's quoted
311     * original — so a message with genuine original content (or a genuine
312     * attachment) still gets published, while a message that's nothing but
313     * quoted/forwarded content with no comment of the sender's own still
314     * gets rejected exactly as before. This is a strict widening: it never
315     * changes what is_reply_or_quote() itself matches, only what happens
316     * once it matches.

319     * caller is expected to have already confirmed is_reply_or_quote() —

351        } while ( $subject !== $before );

353        $subject = str_ireplace( '[agnosis]', '', $subject );
354        $subject = trim( (string) preg_replace( '/\s+/', ' ', $subject ) );

357        foreach ( self::QUOTE_BODY_PATTERNS as $pattern ) {

363            }
364        }
