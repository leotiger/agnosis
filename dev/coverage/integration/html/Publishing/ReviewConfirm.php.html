1<?php
2/**
3 * Frontend shim for email action links.
4 *
5 * Artists receive one-click action links in notification emails.  Previously
6 * those links pointed directly at REST endpoints:
7 *
8 *   /wp-json/agnosis/v1/review/{id}/approve?token=<token>
9 *
10 * Tokens in query strings appear in server access logs, HTTP Referer headers
11 * sent to external resources, and browser history.  This class moves those
12 * tokens off the REST layer:
13 *
14 *   /?agnosis_review=1&id={id}&action=approve&token=<token>   (email link, GET)
15 *       → renders a confirm page with a single POST button (no action taken yet)
16 *       → artist clicks the button → POST /?agnosis_review=1 (token in POST body)
17 *       → server processes via rest_do_request() (no logged REST URL)
18 *       → 302 → /?agnosis_result=approve                      (clean URL)
19 *
20 * The GET/POST split exists because corporate mail-security scanners (Outlook
21 * SafeLinks, Mimecast, Proofpoint, etc.) prefetch links in incoming email to
22 * scan them — issuing a GET, never a POST, and never clicking a button. Before
23 * this split, that prefetch alone was enough to approve, reject, or trash
24 * artwork, or consume a single-use token so the artist's real click showed
25 * "link expired". See docs/security audit §2a.
26 *
27 * The token still appears only once in a URL — in the initial GET's server log
28 * entry. The confirmation POST carries it only in the request body (a hidden
29 * form field), never in a query string, so it is never logged a second time,
30 * never forwarded to the REST access log, never in browser history after the
31 * final redirect, and never in a Referer header (same-origin redirect).
32 *
33 * Final text edits on approve (2026-07-08): the approve confirm page shows the
34 * post's title/excerpt/body as an editable form rather than a plain button.
35 * post_title is always already in the artist's own language (dual-title
36 * design, never AI-translated — see Compat\LinguaForge), so it's shown as-is.
37 * Excerpt/body are AI-authored in the site's primary language at this stage
38 * (audit §3d — "content at rest is always primary-language"), so they are
39 * back-translated for display via SubmissionTranslator::translate_text() — the
40 * same lightweight, single-purpose translate call Notification.php already
41 * uses for the review email preview, deliberately NOT the full generative
42 * writing pipeline that produced the draft ("bypass AI" per the artist's own
43 * framing of this feature).
44 *
45 * That back-translation is cached in `_agnosis_review_backtranslation` post
46 * meta, keyed by a hash of the current excerpt/body and the artist's language.
47 * Without this cache, every GET — including a mail-scanner prefetch, and
48 * every time the artist reopens the same link — would re-spend an AI call;
49 * with it, translation happens at most once per edit generation.
50 *
51 * If the artist changes anything, the edited excerpt/body are translated back
52 * into the primary language on submit (again via translate_text(), not the
53 * generative pipeline) and the post is saved-and-published in one call via the
54 * existing PUT /agnosis/v1/review/{id} route. If nothing changed, the plain
55 * POST /agnosis/v1/review/{id}/approve route runs exactly as before — no
56 * translation call at all in the common case.
57 *
58 * Safeguard: if the submitted title or body is blank after trimming, the whole
59 * approval is cancelled — nothing is published or changed, and the token is
60 * NOT invalidated, so the artist can simply try again from the same email
61 * link. This re-renders the same confirm form inline (never a redirect) so
62 * the token never appears a second time in a URL — the same reasoning that
63 * motivates the GET/POST split above.
64 *
65 * Post types (2026-07-08): the approve form's field set is CPT-aware
66 * (APPROVE_FIELDS) — artwork gets title+excerpt+body, biography gets
67 * title+body (no excerpt concept), event gets body only (title isn't
68 * artist-editable anywhere else in the plugin either) — mirroring
69 * Artist\ContentEditor::EDITABLE_FIELDS's own per-CPT restrictions. This was
70 * fixed alongside ReviewEndpoints's matching artwork-only gate bug, which had
71 * silently 404'd every biography/event approve/reject/save call regardless of
72 * token validity.
73 *
74 * Token check ordering (fourth audit §3a, 2026-07-09): render_approve_confirm()
75 * (GET) and handle_approve_submission() (POST) both call the shared
76 * require_valid_token() gate — ReviewEndpoints::verify_token() — as their very
77 * first step, before any draft content is displayed, any AI translate_text()
78 * call is made, or any post meta is written. Previously the only real token
79 * check on this path was inside the final rest_do_request() dispatch, well
80 * after the GET path had already rendered the draft's title/excerpt/body and
81 * the POST "edited" path had already regenerated and written
82 * _agnosis_translated_title — a guessable sequential post ID plus any string
83 * in the token slot was enough to read an unpublished submission and spend an
84 * unauthenticated AI call.
85 *
86 * Hooks registered in Plugin::register_services() on 'template_redirect' (priority 1).
87 *
88 * @package Agnosis\Publishing
89 */

412                    }
413                }
414            }
415        }

418        $rest_request->set_param( 'token', $token );

429    /**
430     * Shared read-only token gate for the approve confirm form (fourth audit
431     * §3a) — the same `hash_equals()` + expiry check
432     * `ReviewEndpoints::check_access()` already performs for the REST layer,

434     * there is exactly one authoritative implementation rather than a second
435     * copy that could drift out of sync with it.

438     * handle_approve_submission() (POST) — before any draft content is

441     * token already produces later via the REST dispatch, so the visible
442     * behavior for an invalid token is unchanged from the artist's/attacker's
443     * perspective — only how early it is rejected changes. Always exits.

447            $this->redirect_result( 'error', $post_type );
448        }
449    }

451    /**
452     * Resolve an artist's language code (ISO 639-1) from their WP user locale.
453     *
454     * Mirrors the inline conversion Notification::on_post_drafted() already
455     * uses ('es_ES' → 'es'), kept independent of Lingua Forge here — this page
456     * must work whether or not LF is active, same as SubmissionTranslator
457     * itself. Returns '' when the artist has no declared locale (nothing to
458     * back-translate against).

460    private function resolve_artist_lang( int $artist_id ): string {
461        $locale = (string) get_user_meta( $artist_id, 'locale', true );
462        if ( '' === $locale ) {
463            return '';
464        }

468    /**
469     * Return [excerpt, body] in the artist's own language for display on the
470     * approve confirm form, translating and caching on first use.
471     *
472     * Returns the untranslated post fields as-is (no AI call at all) when: the
473     * artist has no declared locale, no AI provider is configured, or the
474     * artist's language already matches the site's primary language — the
475     * common single-language case costs nothing extra.
476     *
477     * @return array{0:string,1:string}
478     */
479    private function get_display_text( \WP_Post $post ): array {
480        $body_plain = wp_strip_all_tags( (string) $post->post_content );
481
482        $artist_lang = $this->resolve_artist_lang( (int) $post->post_author );
483        if ( '' === $artist_lang ) {
484            return [ $post->post_excerpt, $body_plain ];
485        }
486
487        $translator = SubmissionTranslator::from_settings();
488        if ( null === $translator ) {
489            return [ $post->post_excerpt, $body_plain ];
490        }
491
492        $primary_lang = $translator->resolve_target_language();
493        if ( $primary_lang === $artist_lang ) {
494            return [ $post->post_excerpt, $body_plain ];
495        }
496
497        $source_hash = md5( $post->post_excerpt . '|' . $body_plain );
498        $cached      = get_post_meta( $post->ID, self::BACKTRANSLATION_META, true );
499
500        if (
501            is_array( $cached )
502            && ( $cached['hash'] ?? '' ) === $source_hash
503            && ( $cached['lang'] ?? '' ) === $artist_lang
504        ) {
505            return [ (string) $cached['excerpt'], (string) $cached['body'] ];
506        }
507
508        // Not cached, or the source text/artist language changed since the
509        // cache was built — this is the only point in the whole confirm flow
510        // that spends an AI call on a plain GET. Cached immediately below so a
511        // mail-scanner prefetch or a second visit to the same link never
512        // re-spends it.
513        $excerpt_display = '' !== trim( $post->post_excerpt ) ? $translator->translate_text( $post->post_excerpt, $artist_lang ) : '';
514        $body_display    = '' !== trim( $body_plain ) ? $translator->translate_text( $body_plain, $artist_lang ) : '';
515
516        update_post_meta( $post->ID, self::BACKTRANSLATION_META, [
517            'hash'    => $source_hash,
518            'lang'    => $artist_lang,
519            'excerpt' => $excerpt_display,
520            'body'    => $body_display,
521        ] );
522
523        return [
524            '' !== $excerpt_display ? $excerpt_display : $post->post_excerpt,
525            '' !== $body_display ? $body_display : $body_plain,
526        ];
527    }

593    /**
594     * Render the approve confirm form — editable title/excerpt/body, prefilled
595     * in the artist's own language, plus hidden "original" fields used by
596     * handle_approve_submission() to detect whether anything changed.
597     *
598     * @param array<string,string> $prefill Retry values (see render_confirm()).
599     */
600    private function render_approve_confirm( int $id, string $token, array $prefill, string $error ): void {
601        $post = get_post( $id );
602        if ( ! $post instanceof \WP_Post || ! in_array( $post->post_type, ReviewEndpoints::REVIEWABLE_POST_TYPES, true ) ) {
603            wp_safe_redirect( home_url( '/' ) );
604            exit;
605        }
606
607        // Fourth audit §3a: this used to check only that the post exists and is
608        // reviewable — not that the supplied token is the one actually issued for
609        // it, or even that a token exists at all. Since this method renders the
610        // draft's title/excerpt/body (translated into the artist's language via
611        // get_display_text(), which can itself spend an AI call on cache miss),
612        // a guessable sequential post ID plus ANY string in the token slot used
613        // to be enough to read an unpublished submission. Always exits on failure.
614        $this->require_valid_token( $id, $token, $post->post_type );
615
616        // No ?? fallback needed: the gate above already guarantees $post->post_type
617        // is one of ReviewEndpoints::REVIEWABLE_POST_TYPES, and APPROVE_FIELDS has
618        // an entry for each of those three keys — PHPStan proves the offset always
619        // exists here, so an unreachable fallback would just be dead code.
620        $fields      = self::APPROVE_FIELDS[ $post->post_type ];
621        $has_title   = in_array( 'title', $fields, true );
622        $has_excerpt = in_array( 'excerpt', $fields, true );
623
624        // Baseline is always recomputed (cheap — cached after the first call)
625        // so the hidden orig_* fields stay correct even on a validation-error
626        // retry, which must diff against the ORIGINAL back-translation, not
627        // against whatever the artist just (invalidly) typed.
628        [ $baseline_excerpt, $baseline_body ] = $this->get_display_text( $post );
629        $baseline_title = $post->post_title;
630
631        $title   = $prefill['title']   ?? $baseline_title;
632        $excerpt = $prefill['excerpt'] ?? $baseline_excerpt;
633        $body    = $prefill['body']    ?? $baseline_body;
