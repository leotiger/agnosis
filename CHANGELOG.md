## [0.9.15] — 2026-07-10

### Added
- **The event approve confirm form (`ReviewConfirm`) now exposes Date, Hour, Location, Timezone, and Address as individually editable fields before publishing**, instead of only the AI-written description text. Date and Hour are two separate inputs matching how an artist actually thinks about an event, combined back into the existing single `_agnosis_event_date` meta on save (e.g. "2026-08-15T19:00") — every existing reader of that meta (`render_event_date()`, `order_events_archive()`, `ContentEditor`'s post-publish event_date field) is unaffected. Timezone and Address are new fields entirely: `AI\Pipeline::extract_event_fields()` now also extracts them from the artist's email (new `_agnosis_event_timezone`/`_agnosis_event_address` meta), a new `agnosis/event-address` block renders the address on the single event template (paired with `agnosis-theme` 0.5.6), and `render_event_date()` appends the timezone after the formatted date/time when set. Invalid input (an unparseable date, an unrecognised timezone identifier) is silently ignored on approve rather than blocking publication — both remain correctable afterwards via `Artist\ContentEditor`'s existing post-publish correction, which also gained `event_address`/`event_timezone` fields. (`includes/AI/Pipeline.php`, `includes/Publishing/PostCreator.php`, `includes/Artist/Profile.php`, `includes/Artist/ContentEditor.php`, `includes/Publishing/ReviewConfirm.php`)
- **The biography approve confirm form now has an editable "Portfolio link" field**, so a misspelled or otherwise-rejected portfolio URL from the join application can be corrected before publishing, not just after. Correcting it re-runs `EmbedPolicy::is_allowed()` against the new URL and rebuilds the `wp:embed` block accordingly. As a side effect, this also fixes a latent bug where editing ANY part of a biography draft's body on the approve form silently dropped an already-approved portfolio embed entirely — `ReviewEndpoints::save()`'s content rebuild only ever preserved leading image/gallery blocks, never a trailing `wp:embed`. The embed is now always re-synced on approval (cheaply, without re-running `EmbedPolicy` unless the URL actually changed), regardless of what else was edited. (`includes/Publishing/ReviewConfirm.php`, `includes/Artist/ApplicationBiography.php`)

### Fixed
- **Any link that didn't clear `EmbedPolicy` — a join application's portfolio URL, or a link an artist mentioned in an artwork/biography/event email — was dropped with nothing beyond a log line, for every content type, not just biography.** `EmbedPolicy::is_allowed()` gained a companion `last_reason()` (non-breaking — every existing `is_allowed()` call site is untouched) that explains WHY a given link was rejected (not on the trusted-platform list with AI review off, the page couldn't be safely fetched, AI review was inconclusive, or AI review flagged the content). Both drafting paths now record every dropped link plus that reason in the same shape — a new `_agnosis_dropped_links` post meta (JSON array of `{url, reason}`), written by `PostCreator::build_external_link_embeds()` for artwork/biography/event email submissions and by `ApplicationBiography::on_artist_admitted()` for a join application's portfolio field. `Notification::build_email()` now reads this one meta key regardless of post type, listing every dropped link with its specific reason in the review email — superseding the biography-only notice from earlier in this same version, which only ever explained biography's portfolio field and never gave a reason. Biography's approve-form correction (above) also keeps this meta in sync when the artist fixes the link. (`includes/Publishing/EmbedPolicy.php`, `includes/Publishing/PostCreator.php`, `includes/Artist/ApplicationBiography.php`, `includes/Publishing/Notification.php`, `includes/Publishing/ReviewConfirm.php`)
- **The "new application, please vote" email sent to every existing artist was the one remaining plain-text email in the plugin**, unstyled next to the acknowledgment/welcome/review emails that already used the shared branded HTML template. `AdmissionNotification::send_vote_email()` now builds its body via a new `build_vote_email_body()` (same `EmailBranding` header, 600px card, and styled Yes/No buttons as `build_acknowledgment_body()`/`build_welcome_body()` in the same file) and sends with `html_headers()` instead of `text_headers()`, using `EmailFooter::html()`/`edit_reminder_html()` in place of their plain-text counterparts. Same content as before (bio/portfolio/statement, vote links, footer, edit reminder) — styling only. (`includes/Artist/AdmissionNotification.php`)
- **The artist-breadcrumb's "Biography"/"Events" links could overflow or wrap awkwardly on mobile, next to the artist's name, in languages whose translations of those words are long.** Both links are now icon-only — a book icon for Biography, a calendar icon for Events — a fixed width regardless of locale. The translated word moves to the link's `aria-label` (screen readers) and `title` (hover tooltip) instead of visible text, and the "|" separator between them is gone entirely, replaced by consistent spacing that works whether one or both links are present. Paired with theme-side icon styling (`agnosis-theme` 0.5.7). (`includes/Network/SubdomainNavigation.php`, `blocks/artist-breadcrumb/block.json`)
- **The Community announcement address (`agnosis_email_community`) was missing from the "here's every address you can use" quick-reference card that appears in every work-related notification email** — an artist reading any of those emails had no way of finding out that address existed unless they'd dug through Settings themselves. `EmailFooter::address_options()` now includes it alongside Artwork/Biography/Events/Replace/Remove/Promote/Photo-only/Pure, so it now shows up everywhere that list already does (welcome, acknowledgment, and every review/admission email built on `EmailFooter::html()`/`plain_text()`). The "goodbye" self-removal address remains deliberately excluded — that's a departure action, not something you'd send mail to as part of participating. (`includes/Core/EmailFooter.php`)
- **The Promote address's description ("mark an artwork as featured in your gallery") was ambiguous about which gallery it affects.** An artist's own subdomain already shows every one of their published pieces — promote@ actually decides which single piece represents them in the shared, cross-artist gallery on the main site (`blocks/gallery-overview`'s per-artist featured-post lookup), confirmed still working exactly as originally specified: a per-artist budget that scales inversely with the total number of active artists (more artists → fewer slots each, down to one), a promoted piece filling that artist's slot first, and their most recent piece filling the rest. The address description (Settings → Email, the shared "every address" reference card, and the welcome email's alias list) now says "the shared community gallery on the main site" explicitly, instead of the ambiguous "your gallery." No behavior changed — wording only. (`includes/Core/EmailFooter.php`, `includes/Admin/Settings.php`, `includes/Artist/AdmissionNotification.php`)
- **The footer's newsletter-subscribe bell and the Twitter/Facebook/Pinterest icons next to it rendered too small by default.** `Newsletter\PopoverBlock`'s trigger icon is now 26×26 (was 18×18) and its close-panel icon 20×20 (was 14×14); the shared `.lf-icon-btn` box both sit in grew from 2.25rem to 3rem to match. Paired with a theme-side change (`agnosis-theme` 0.5.8): the footer's Social Icons block now explicitly sets `"has-large-icon-size"` instead of relying on the (smaller) unstyled default, so it stays visually flush with the bigger bell button next to it. (`includes/Newsletter/PopoverBlock.php`, `blocks/newsletter-popover/frontend.css`)

## [0.9.14] — 2026-07-10

### Fixed
- **`agnosis/artist-breadcrumb` pipe-separated the artist's name from "Biography" even when no Events link followed it, and left no way for a theme to lay the name and the Biography/Events links out on opposite sides.** `SubdomainNavigation::render_block()` now wraps the artist's name and the Biography/Events links in two separate spans (`.agnosis-artist-breadcrumb__name` and `.agnosis-artist-breadcrumb__links`) instead of one pipe-joined string — the name is never pipe-separated from anything, and "Biography"/"Events" are pipe-separated from each other only when both are present. Paired with a theme-side layout change (`agnosis-theme` 0.5.5) that puts the two spans on opposite sides of the block (name on the reading-start side, links on the reading-end side, auto-reversing on RTL). (`includes/Network/SubdomainNavigation.php`, `blocks/artist-breadcrumb/editor.js`, `blocks/artist-breadcrumb/block.json`)
- **The "Biography" link in the breadcrumb didn't follow the visitor's current language, unlike the artist-name link right next to it.** `biography_permalink()` queried `agnosis_biography` posts by author only, with no language scoping — since Lingua Forge translation fan-out creates one additional `agnosis_biography` post per translated language, `get_posts()`'s date-ordered result could return any of those siblings (e.g. always the most recently translated one), regardless of which language the page itself was being viewed in. The lookup now scopes to the artist's source-language post first, then resolves it to the visitor's own language's published sibling via a new `localized_post()` helper — the same "look up by `_lf_lang`, then swap via `linguaforge_get_translations()`" pattern `Newsletter\Digest::localized_post()` already used. No-ops (falls back to the single post found) when Lingua Forge isn't active. (`includes/Network/SubdomainNavigation.php`)

## [0.9.13] — 2026-07-09

### Added
- **Template safeguard for translated posts, on Lingua Forge >= 2.6.1.** Lingua Forge 2.6.1 fixed a bug where `TranslationTrigger::create_translated_post()` — the function Agnosis's own translation trigger (`request_translations()`) always goes through — never assigned a translated post its language-specific FSE template (`_wp_page_template`), even when a matching `single-{post_type}-{lang}` template already existed in the database (e.g. one scaffolded via Lingua Forge's own Settings → Router FSE tools). LF 2.6.1 also shipped `linguaforge_sync_templates( $post_id, $check_caps )`, a free, non-AI, content-untouched call that re-resolves and re-writes the correct template across every sibling in a translation group — documented by LF as "safe to run repeatedly." New `Compat\LinguaForge::sync_translated_template()`, hooked on `linguaforge_translation_complete` only when `LINGUAFORGE_VERSION >= 2.6.1`, calls it (with the source/primary post ID, `$check_caps = false`) after every translation completes. This is additive defense-in-depth on top of LF's own fix, not a workaround for its absence — it protects against future template drift from a theme change, a template rename, or a future LF regression, at no ongoing cost. No-ops entirely on any LF version before 2.6.1. (`includes/Compat/LinguaForge.php`)

### Tests
- **`LinguaForgeCompatTest`** gained coverage for the safeguard above: `sync_translated_template()` calls `linguaforge_sync_templates()` with the source post's ID (not the translated post's) and `$check_caps = false`; it's a no-op for a non-Agnosis source post type; and — mirroring the existing `copy_translated_meta()` version-gate regression test's own documented caveat about `LINGUAFORGE_VERSION` being fixed at `'1.0.0-test'` for the whole test process — a new test proves the negative case directly testable under that fixed stub: the hook is NOT registered while `LINGUAFORGE_VERSION` is below 2.6.1. New `linguaforge_sync_templates()` global stub added alongside the existing `linguaforge_trigger_translation()`/`linguaforge_queue_translation()` stubs. (`tests/php/Integration/Compat/LinguaForgeCompatTest.php`, `tests/php/Integration/Compat/Stubs/lf_global_stubs.php`)

## [0.9.12]

### Fixed
- **`remove@` and `promote@` failed completely silently on a title mismatch** — fifth audit's headline finding (§2b). `PostCreator::handle_removal_request()`/`handle_promotion_request()` used to log a warning and return on a subject that matched no post, after which `handle()` marked the queue row `'published'` regardless — the artist got no email at all, and the Inbox admin table showed a misleading "processed" status for a request that did nothing. Both methods now return a bool the caller uses to mark the queue row `'skipped'` (with a descriptive reason) instead of `'published'` on a miss, and both fire a new artist-facing action — `agnosis_removal_target_not_found` / `agnosis_promotion_result` — consumed by new `Notification::on_removal_target_not_found()`/`on_promotion_result()` methods that send a localized "we couldn't find that title" email listing the artist's current titles. `promote@` also gets a *success* confirmation email for the first time — previously even a working promotion was silent. (`includes/Publishing/PostCreator.php`, `includes/Publishing/Notification.php`, `includes/Core/Plugin.php`)
- **Exact-title matching on `remove@`/`promote@` had no tolerance for typos or reworded titles, unlike the fuzzy detection artwork intake already enjoys** — fifth audit §2c. New `PostCreator::gather_title_context()` reuses the same cheap AI text-comparison pattern `find_duplicate_post()`'s third layer already uses to suggest which existing title an artist most likely meant, surfaced as a "did you mean X?" line in the new not-found email above. The suggestion is never acted on automatically — for `remove@` specifically, since the real consent step is the confirmation-link click (not the request email), a removal token is safely pre-generated for the suggested post so the email can include a ready-to-use confirm link; a wrong guess simply dies unclicked. `promote@` has no confirmation step to attach a token to, so its miss email is suggestion-only text. (`includes/Publishing/PostCreator.php`, `includes/Publishing/Notification.php`)
- **The subject-line `[Indicator]` fallback (biography/event/photo/pure) only recognized English keywords** — fifth audit §2d. Address-based routing is the primary mechanism, but the bracket-keyword fallback exists precisely for artists whose mail situation is odd — the least technical cohort — and a `[Biografía]` or `[活动]` silently became a plain artwork submission. `PostCreator::INDICATORS` now also recognizes the same four indicators translated into the 17 locales this plugin ships template-string catalogs for (see `languages/*.po`) — a static keyword table, no AI, no settings. (`includes/Publishing/PostCreator.php`)
- **Alias routing (`remove@`/`promote@`/`bio@`/`event@`/etc. and the `goodbye@`/`community@` special lanes) only ever matched the first `To:` address, on both intake transports** — fifth audit §5a. An artist writing to `remove@` while CCing a friend, or whose mail client serialized `To:` in an unexpected order, had their message silently fall through to the plain artwork pipeline and bounce a confusing "no attachment" reply. `Email\Parser` now collects every `To:`/`Cc:` recipient (`to_addresses`) on both the IMAP and webhook paths, `PostCreator::resolve_post_type()` matches any configured alias against the full list, and `Email\Inbox`/`Email\Webhook`'s `goodbye@`/`community@` detection does the same. (`includes/Email/Parser.php`, `includes/Email/Inbox.php`, `includes/Email/Webhook.php`, `includes/Publishing/PostCreator.php`)
- **The `reject`/`remove` email-link confirm page rendered for any token, valid or not** — fifth audit §2e. Unlike `approve` (fourth audit §3a), `ReviewConfirm::render_confirm()`'s reject/remove branch showed the generic "are you sure?" page regardless of token validity — an artist with an expired or already-used link only found out after clicking the button and hitting the REST layer's own check. New `require_valid_action_token()` calls the same check one step earlier, dispatched to the right token store per action (`ReviewEndpoints::verify_token()` for reject, `RemovalEndpoints::verify_token()` — now `public static`, mirroring `ReviewEndpoints` — for remove). (`includes/Publishing/ReviewConfirm.php`, `includes/Publishing/RemovalEndpoints.php`)
- **A signed webhook request (Mailgun `timestamp`/`token`/`signature`, or the generic `X-Agnosis-Signature` scheme) validated forever and could be re-submitted at will** — fifth audit §3a. `Webhook::verify_signature()` checked only that the HMAC matched, never that the signed request was recent or hadn't been seen before — anyone who ever captured a valid signed payload (proxy logs, a misconfigured CDN, the artist's own outbox if they run the ESP) could replay it indefinitely, each replay re-running the full AI pipeline at platform cost. Separately, `Webhook::handle()`'s queue UID hashed the Message-Id together with `time()`, so even a replay of the exact same signed request produced a different UID and defeated the queue table's own `UNIQUE KEY (message_uid)` idempotency guard, meaning every replay was a genuine duplicate submission, not a harmless no-op. New `check_replay_freshness()` rejects a signed request whose timestamp is missing, malformed, or more than 5 minutes stale, and remembers each specific signed request (Mailgun's `token`, or the signature itself for the generic scheme) for 10 minutes via a transient so it can't be replayed twice inside that window — applied to both signing schemes (the generic scheme's replay protection is opt-in via a new `X-Agnosis-Timestamp` header; senders not yet sending it fall through to the original body-only check, unchanged). The queue UID now hashes only the Message-Id (falling back to a hash of the parsed submission when no Message-Id is present), and the insert itself now uses `INSERT IGNORE` + a follow-up `SELECT`, mirroring `Email\Inbox::enqueue()`'s existing idempotent-insert pattern for the IMAP transport — so a retry or replay that does slip through the freshness window now collides harmlessly with the original row instead of enqueueing a second time. (`includes/Email/Webhook.php`)
- **The webhook `goodbye@` alias skipped the admitted-artist check the IMAP path enforces, and webhook `goodbye@`/`community@` events left no trace in the Inbox admin table** — fifth audit §3b/§5a. `Webhook::handle()`'s goodbye branch only checked that the sender resolved to *some* WP user before calling `Departure::initiate_removal_for_user()` — `Inbox::handle_goodbye_email()`'s IMAP-path equivalent requires `Admission::is_admitted_artist()` first. Exposure was bounded (`initiate_removal_for_user()` itself still requires an active `agnosis_applications` row), but a registered, non-admitted account (a pending applicant, a banned artist, a plain subscriber on a shared install) could reach `Departure` via webhook and not via IMAP. Separately, `Inbox` records every goodbye@/community@ outcome as a reason-keyed queue row via `mark_no_artwork()`, so the Inbox admin table shows the full story on IMAP sites; `Webhook` only ever returned a REST response, so the same events on a webhook site were visible solely in the Logs tab. The goodbye branch now gates on `Admission::is_admitted_artist()` exactly like the IMAP path, and both the goodbye@ and community@ branches now call new `Webhook::mark_alias_event()` — a queue-row recorder mirroring `Inbox::mark_no_artwork()`'s reason-label/status-override tables, keyed by a deterministic `webhook-alias-` + Message-Id UID — at every branch exit, so the same reasons (`goodbye_handled`, `goodbye_no_membership`, `goodbye_throttled`, `goodbye_non_artist`, `community_handled`, `community_throttled`, `community_empty`, `community_auto_submitted`, `community_too_long`, `community_non_artist`) now populate the Inbox admin table regardless of transport. The webhook community branch's not-admitted reason key was also renamed from `community_not_admitted` to `community_non_artist` to match the IMAP path's reason exactly. (`includes/Email/Webhook.php`)
- **A purely numeric tag or medium name (e.g. a literal year like "2026", or an AI translation that happened to come back as a bare number) could silently mis-assign to an unrelated, pre-existing term instead of the correctly-named one when `Compat\LinguaForge::sync_taxonomy()` copied translated terms onto a translated post** — fifth audit §6, pre-existing since 0.9.9. WordPress's own `term_exists()` (called both directly by `sync_taxonomy()` and internally by `wp_set_object_terms()`) adds a `t.term_id = %d` fallback to its lookup SQL whenever the term string is numeric-looking, so a translated name of "2026" could match whatever unrelated term happened to have term ID 2026 rather than a term actually named "2026" — and the same ambiguity meant such a term could dodge the `TRANSLATED_TERM_META` flag §4c added, since `term_exists()`'s false-positive match made it look pre-existing when it wasn't. New `sync_taxonomy()` logic resolves any numeric-looking translated name to a real term ID itself (via `get_term_by()`/`wp_insert_term()`, new `resolve_numeric_term_name()`) before it ever reaches `term_exists()`/`wp_set_object_terms()` as an ambiguous string — passing a genuine int sidesteps the ambiguity entirely, since WordPress's own lookup only does an exact ID match for a real int. Non-numeric names are completely unaffected. (`includes/Compat/LinguaForge.php`)

### Changed
- **`Pipeline::process()` ran the full ~600-word editorial describe() prompt (high-detail vision, 1500 max output tokens) on every attachment in a gallery submission, but only the first successfully-described attachment's title/excerpt/body/medium is ever published** — fifth audit §4c. `Publishing\PostCreator::primary_result()` picks the first result with `description_ok = true` and discards every other attachment's own title/excerpt/body/medium entirely; the rest only ever contribute their alt text (per attachment), tags (merged into the post's tag list), and photo-quality score (the per-image quality-rejection gate) — so a 5-image gallery submission spent 5 full editorial prompts to use only 1 of them. New `ProviderInterface::describe_secondary()` — implemented in `Anthropic`, `OpenAI`, and `WordPressAI` — asks for exactly the three fields actually consumed from a non-primary result (alt text, tags, photo quality) via a fixed, few-dozen-word prompt instead of the full customizable one, with a 300 max-token ceiling instead of 1500. `Pipeline::process()` now tracks whether a primary result has already been found (matching `primary_result()`'s own "first success wins" logic exactly) and only calls the full `describe()` until that happens — every attachment after that point uses `describe_secondary()` instead. If the first attachment's full description fails, the next attachment still gets the full prompt (a primary hasn't been found yet), so the existing resilience — any one image's description failure doesn't sink the submission as long as a later one succeeds with real title/excerpt/body — is fully preserved; only once a real primary is confirmed do later images downgrade to the slim prompt. Image tokens are unaffected (`detail: 'high'` unchanged) — the saving is entirely on the ~2,000+ text tokens per secondary image, not the vision input itself. `WordPressAI::describe_secondary()` returns an immediate failure at zero API cost, since that provider is text-only and was never able to genuinely analyse a secondary image's pixels regardless. (`includes/AI/ProviderInterface.php`, `includes/AI/PromptConfig.php`, `includes/AI/Providers/Anthropic.php`, `includes/AI/Providers/OpenAI.php`, `includes/AI/Providers/WordPressAI.php`, `includes/AI/Pipeline.php`, `includes/Publishing/PostCreator.php`)
- **Community broadcasts translated the subject and body once per recipient instead of once per language** — fifth audit §4a, the plugin's single largest recurring AI-cost saving. `CommunityBroadcast::broadcast()` called `translate_text()` (subject + body) inside the per-recipient loop, so twenty Spanish-speaking members on one broadcast meant twenty separate, byte-identical Spanish translations. Recipients are now grouped by target language first; each distinct language is translated exactly once and every recipient in that group reuses the same copy (N recipients × 2 calls → L languages × 2 calls — on a 50-member, 5-language community that's 100 calls down to 10 per broadcast). `switch_to_locale()` still runs per recipient, not per group — that's the template chrome in `send_one()` ("Subject:", "Hit reply…"), which reads WP's full locale (e.g. `es_ES` vs `es_MX`), a finer distinction than the 2-letter language code translation is grouped by. No feature change: recipients sharing a language already received identical wording before this fix, since a "broadcast" implies exactly that. (`includes/Artist/CommunityBroadcast.php`)
- **The review email spent three separate, uncached translation calls per cross-language draft, then the confirm page re-translated the same content into a different cache** — fifth audit §4b. `Notification::on_post_drafted()` back-translated title, excerpt, and an 80-word body preview in three sequential `translate_text()` round trips, each carrying its own prompt envelope and none of it cached; when the artist clicked through, `ReviewConfirm::get_display_text()` translated the excerpt and full body *again* into a separate `_agnosis_review_backtranslation` cache. New `SubmissionTranslator::translate_fields()` batches an arbitrary named set of fields into one JSON-envelope `chat()` call (generalising the existing subject+body pattern `call_translate()` already used at intake); `on_post_drafted()` now translates title, excerpt, and the *full* body in that single call, derives the email's 80-word preview by trimming the translated body, and writes the translated excerpt/full-body straight into `ReviewConfirm::BACKTRANSLATION_META` (now `public`, same hash/lang scheme `get_display_text()` already reads) — so the artist's click-through from that same email almost always cache-hits instead of translating a second time. Net effect: up to ~5 translation calls per cross-language draft down to 1, occasionally 2 on a cache miss. (`includes/AI/SubmissionTranslator.php`, `includes/Publishing/Notification.php`, `includes/Publishing/ReviewConfirm.php`)
- **Every artwork publish translated its title once per enabled site language, in separate round trips** — fifth audit §4d. `Compat\LinguaForge::build_title_translations()` looped `SubmissionTranslator::translate_text( $title, $lang )` once per target language, each call re-shipping a full translate prompt envelope for a ~10-word string — a 5-language site meant 5 separate AI calls per artwork published. New `SubmissionTranslator::translate_to_languages()` translates the title into every target language in one `chat()` call, returning a language-code-keyed result — the same JSON-envelope pattern `translate_fields()` (§4b) already established, generalised to the opposite axis (one field, many languages, instead of many fields, one language). `build_title_translations()` now calls it once instead of looping; the identical "only keep an actual change" filter the old per-language loop applied is preserved inside the new method. (`includes/AI/SubmissionTranslator.php`, `includes/Compat/LinguaForge.php`)

## [0.9.11] — 2026-07-09

### Fixed
- **Re-translation never refreshed a translated sibling's gallery images, thumbnail, event date/location, or AI display title on Lingua Forge >= 2.4.0** — the fourth audit's headline finding (§4b). `Compat\LinguaForge::__construct()` forked on LF version: on LF >= 2.4.0 it registered only `supply_translated_meta()` on LF's "born-with" filter, `linguaforge_translated_post_meta`, which (verified against the LF 2.6.0 source) fires **only** in `TranslationTrigger::create_translated_post()` — the re-translation path, `update_translated_post()`, applies no meta filter at all. So once a translated sibling already existed, nothing ever ran `copy_translated_meta()` again: a `replace@` photo swap left the old images live on every translated page, a front-end event date/location correction never reached translated siblings, and an artwork title edit's corrected `_agnosis_translated_title` never refreshed on siblings either — despite `ContentEditor::propagate_title()`'s own docblock promising exactly that refresh. `copy_translated_meta()` is now also registered on `linguaforge_translation_complete` **unconditionally**, alongside (not instead of) the born-with filter — that action fires on both creation and re-translation at every LF version, and the method is a plain, idempotent `update_post_meta()` re-copy, so on first creation it merely rewrites the values the born-with filter already supplied. (`includes/Compat/LinguaForge.php`)
- **The approve confirm page ran side effects before checking the review token at all** — fourth audit §3a. Two halves, both introduced by the 0.9.9 editable-form feature: (a) `ReviewConfirm::render_approve_confirm()` (the GET page an artist reaches from their review email) displayed the draft's title/excerpt/full body — including an AI back-translation call into the artist's language on cache miss — to anyone who supplied a valid post ID and ANY string as the token, since the token itself was never actually checked before rendering; (b) `handle_approve_submission()` (the POST from that page) would, for an artwork whose title changed, call the AI translator and write `_agnosis_translated_title` to the post **before** the only real token check on this path, which happened later inside the final `rest_do_request()` dispatch — so a forged POST with a garbage token could already both spend an AI call and overwrite the publicly visible display subtitle of any reviewable post. `ReviewEndpoints::verify_token()` (the exact `hash_equals()` + expiry check the REST layer already performs) is now `public static` and called by both methods as their very first step, before any content is shown, any AI call is made, or any meta is written; on failure both redirect to the existing `?agnosis_result=error` page, the same outcome a bad token already produced later via the REST dispatch. (`includes/Publishing/ReviewConfirm.php`, `includes/Publishing/ReviewEndpoints.php`)
- **`community@`/`goodbye@` bypassed the opt-in SPF/DKIM auth gate entirely, and `goodbye@` had no per-sender throttle on either intake path** — fourth audit §3b. Both `Inbox::poll()` and `Webhook::handle()` routed the two aliases before the `agnosis_require_email_auth` check (which sat ~40–90 lines further down, reached only by messages falling through to the normal artwork/bio/event pipeline) — so with auth enforcement turned on, a spoofed From claiming to be a real admitted artist could still relay a `community@` broadcast to every other member under that artist's own name, or trigger a `goodbye@` self-removal confirmation email, an unlimited number of times on the goodbye path (no throttle existed there at all, on either transport). The auth check is now evaluated first, before any alias routing, on both paths; a new **Self-removal (goodbye) requests per sender per day** setting (`agnosis_goodbye_request_limit`, default 3, Settings → Community) throttles `goodbye@` the same way `community@` was already throttled, using a shared `'goodbye_request'` rate-limiter bucket between the IMAP and webhook transports (mirroring how `'community_broadcast'` is already shared between them). (`includes/Email/Inbox.php`, `includes/Email/Webhook.php`, `includes/Admin/Settings.php`)

### Added
- **Tags and medium are now translated onto every translated post, instead of arriving with neither.** Lingua Forge only ever copies `post_title`/`post_content`/`post_excerpt` (plus whatever `linguaforge_translated_post_meta` supplies) onto a translated post — it has no generic taxonomy-term translation or copying logic anywhere in its own code, so a Spanish translation of an English artwork previously published with zero tags and no medium at all, regardless of what the English original had. `Compat\LinguaForge::sync_translated_terms()` now hooks `linguaforge_translation_complete` (fired by LF on both first translation and re-translation) and, for every translated Agnosis post, translates and assigns the source post's `post_tag` terms; for `agnosis_artwork` specifically, it does the same for `agnosis_medium`. - **The Inbox admin page could mislabel a legitimate, registered artist's submission as "unregistered sender"**, and separately show a false "not admitted" rejection for an artist who plainly was admitted — both traced to `Inbox::mark_no_artwork()` discarding the sender's `artist_id` and collapsing every skip reason (unregistered, not-admitted, throttled, auth-failed, no-attachments, goodbye variants) into one generic message. `mark_no_artwork()` now takes and persists the real `artist_id` plus a specific `$reason` (new `SKIP_REASONS` map), threaded through every call site, so the admin log always reflects what actually happened.
- **A multi-image gallery submission's quality gate only ever evaluated the first photo** — a second or third image with a low quality score in the same email was silently kept (or, worse, a bad first image rejected the whole submission even when the other photos were fine). `PostCreator::apply_quality_gate()` now scores every image independently: images below the threshold are dropped individually and logged; the whole submission is only rejected when every image fails.
- **The application-received and application-declined emails showed a footer of work-submission addresses** (artwork@, bio@, replace@, etc.) that don't apply yet — the recipient hasn't been admitted and has no way to use them. Both emails no longer include that footer; it still appears, correctly, on every email sent to an existing artist.

### Added
- **Configurable "what happens next" page after a successful join-form submission.** New Settings → Network field, "After applying, send artists to" (`agnosis_join_success_url`) — when set, a successful application redirects the artist to that page instead of just showing an inline confirmation message; blank keeps the previous inline-only behavior.
- **Artists are now notified by email when a submission is silently dropped for having no usable file** — previously both failure modes (a photo pasted/inlined instead of properly attached, and every attachment failing to convert into something the pipeline can use) left the artist with no idea their email never went anywhere. New `agnosis_submission_no_attachment` action, fired from `Inbox` and from `PostCreator::handle()`, and a new `Notification::on_submission_no_attachment()` email explaining both possible causes and how to resend correctly.
- **HEIC/HEIF support — the default photo format on modern iPhones.** Previously silently rejected at intake. `Email\Parser` now allowlists `image/heic`, `image/heif`, and their `-sequence` (burst/Live Photo) variants; new `MediaAdapter::adapt_heic()` converts to JPEG via Imagick (primary frame only for a sequence) before anything reaches the AI vision pipeline, which doesn't accept HEIC/HEIF natively. Degrades gracefully — via the same `extension_loaded('imagick')` guard already used for PDF rasterization — when Imagick isn't installed, or when it is but its ImageMagick build lacks the `libheif` delegate (a common partial-install case); either way the artist gets the same "couldn't process your photo" notification above rather than a silent drop.
- **The application-received acknowledgment and welcome/admitted emails are now styled HTML**, matching every other outgoing email's branded card layout (`EmailBranding::header_html()`, consistent header/body/footer chrome) instead of plain text.

### Changed
- **The welcome email no longer leads with a password-reset link as the main call to action.** Working with Agnosis — submitting, editing, or removing work — is entirely email-based and needs no WordPress login at all. The email now mentions password recovery only as a low-key, opt-in note at the very end, pointing at WordPress's own self-service "Forgot your password?" flow (`wp_lostpassword_url()`) for artists who want the optional online features (like previewing a submission before it publishes) — rather than baking in a single-use, pre-generated reset-key link that implied logging in was part of the normal flow.
- **Artwork submission authentication clarified**: an inbound submission is identified purely by matching the message's From address against an admitted artist's WordPress account email (`get_user_by( 'email', ... )`) — there is no token, signature, or secret in the loop. Optional SPF/DKIM verification (Settings → Email, `agnosis_require_email_auth`) and per-sender throttling (`RateLimiter::check_sender()`) sit in front of that match as opt-in hardening against a spoofed From header, but neither is required by default. Documented as part of this session's review of the intake flow; no code change was required, since this was already the intended design.

## [0.8.1] — 2026-07-06

### Fixed
- **A newsletter issue could get stuck showing "Sending…" (Send Now disabled) forever, even well past the next scheduled cron tick.** `QueueProcessor::process()`'s per-recipient send loop had no exception handling — if `send_one()` threw for any single recipient (a bad locale render, a branding/image error building that particular email, etc.), the uncaught error killed the entire cron tick before reaching `reconcile_sending_issues()`, which runs once per tick after the loop. That meant *every* currently-"sending" issue, not just the one that errored, silently stopped being reconciled — indefinitely, since the exact same failure recurred on every subsequent tick before ever reaching the reconcile step. `send_one()` is now wrapped in a try/catch per recipient: a failure is logged and that row is retried/eventually marked terminally 'failed' exactly as a normal `wp_mail()` failure already was, and reconciliation always runs regardless of what happened to any individual recipient.
- **The Newsletter Status dashboard (Settings → Newsletter) now self-heals a stuck "Sending…" status the moment an admin views the page**, instead of only reflecting reality on WP-Cron's next tick. On a low-traffic install with no real system cron wired to `wp-cron.php`, that tick may not happen again for a long time. `QueueProcessor::reconcile_sending_issues()` is now public and called at the top of `render_newsletter_dashboard()` — cheap (a few `COUNT` queries), a no-op when nothing is actually "sending".

## [0.4.3] — 2026-07-04

### Changed
- **Artist subdomains no longer replace the visible site title with the artist's name.** `SubdomainRouter` filtered `option_blogname` since 0.1.9 so `wp:site-title` showed the artist rather than "Agnosis" — in practice this left the header with no visible Agnosis branding at all and, combined with `wp:site-title`/`wp:site-logo` both linking to `home_url()` (which `rewrite_home()` already points at the artist subdomain), no way to navigate back to the main portal from an artist's page. The `option_blogname` filter is removed; `document_title_parts`/`wp_title` (the HTML `<title>`, used for the browser tab/SEO) still show the artist's name and are unaffected. The actual "no way back" problem is fixed on the theme side: agnosis-theme's `functions.php` now points the Site Logo and Site Title links at the main Agnosis domain (`agnosis_base_domain` option) instead of `home_url()` whenever `SubdomainRouter::is_artist_subdomain()` is true.

### Fixed
- **Email action links executed on a plain GET — corporate mail-security scanners could trigger them.** Every one-click link in the platform (`ReviewConfirm` review approve/reject/remove, `VouchConfirm` admission votes, `SubscriptionConfirm` newsletter confirm/unsubscribe for both the public list and artists) mutated state the instant its URL was fetched, at `template_redirect`. Link-scanning mail security layers (Outlook SafeLinks/ATP, Mimecast, Proofpoint, and similar) prefetch GET links in incoming email to scan them — which is indistinguishable, server-side, from the recipient actually clicking. A prefetch alone could consume a single-use review token (the artist's real click then shows "link expired or already used"), silently approve/reject/remove artwork, cast a community admission vote on the artist's behalf, or confirm/unsubscribe a newsletter recipient who never opened the email client. Fixed by splitting every one of these shims into two phases: a GET now only renders a themed confirm page naming the action, with a single POST button; the token travels in that form as a hidden field (never in the form's `action` URL, so it is not logged a second time), and the action itself — REST dispatch, vote recording, or subscriber status change — only runs once that POST arrives. Scanners issue GETs and never submit forms, so prefetching a link is now inert. `VouchConfirm` and `SubscriptionConfirm` also had their response pages moved from raw `echo`+`exit` onto `wp_die()` (matching `ReviewConfirm`'s existing pattern), which incidentally closes a testing gap noted in both suites and lets the whole GET/POST split be verified end-to-end; `VouchConfirm`'s error page also now correctly returns HTTP 400 (previously always 200), consistent with the other two shims.
- **The public newsletter signup block never sent the visitor's language, so per-locale rendering never applied to public subscribers.** `Subscription::register_routes()` has accepted a `language` param since 0.4.0 and correctly maps it to a WP locale via `Admission::iso_to_wp_locale()`, but the signup block's `frontend.js` only ever sent `{ email }` — there was no language field, and no other source for the value. Every public-list subscriber therefore had `locale = NULL`, silently resolving to the site default locale at send time and leaving 0.4.1's per-locale digest/intro rendering exercised only by the artist audience in practice. Fixed by having `frontend.js` send `document.documentElement.lang` (the primary subtag, lower-cased — `"es-ES"` becomes `"es"`, the same split-on-hyphen normalization `Admission::apply()` already applies to its `Accept-Language` fallback) as `language` in the subscribe payload. WordPress and Lingua Forge already set `<html lang="…">` per language version of a page, so a visitor subscribing from `/es/` or an `es` subdomain now gets the right locale for free, with no new form field and no server-side change.
- **The community removal-vote email link was completely dead — no handler was ever registered for it.** `DepartureNotification::removal_vote_url()` has generated `?agnosis_removal_vote=1&rid=…&vid=…&vote=…&token=…` links in removal-vote notification emails since the removal-vote feature shipped, but no `template_redirect` handler ever consumed that query string, and there was no other UI to cast a removal vote at all — every link 404'd (or rendered the homepage), silently disenfranchising the entire community on every removal vote. Fixed by adding `RemovalVoteConfirm`, a new email-link shim mirroring `VouchConfirm`'s stateless-HMAC-token pattern and its GET/POST split from day one (a GET renders a themed confirm page; the vote is only recorded on POST). `Departure::cast_vote()` was refactored to delegate to a new shared `record_vote_on_request()` method so the REST endpoint and the email-link shim share one validation path, the same relationship `Admission::record_vote()` already has with `VouchConfirm`.
- **A failed newsletter send was given up on after a single `wp_mail()` failure, with no retry.** `QueueProcessor::process()` marked a queue row `failed` the instant `wp_mail()` returned `false` — a transient SMTP hiccup or a momentary host outbound-rate-limit rejection permanently dropped that recipient from the issue, with no second chance and no visible distinction from a genuinely bad address. Fixed with a bounded retry: `agnosis_newsletter_queue` gains an `attempts` column, and a failed send now stays `pending` (so the next 5-minute cron tick retries it) until three attempts have failed, at which point it is marked terminally `failed` and `resolved_at` records the final-failure time. Existing installs get the column via a guarded `ALTER TABLE` in `Activator::maybe_upgrade()`.
- **No RFC 8058 one-click unsubscribe.** Newsletter mail set `List-Unsubscribe:` but not the paired `List-Unsubscribe-Post: List-Unsubscribe=One-Click` header, and the unsubscribe URL only handled GET — falling short of the RFC 8058 one-click flow that Gmail/Yahoo bulk-sender rules require above ~5k messages/day and that also feeds Gmail/Outlook's built-in "Unsubscribe" affordance and spam-folder placement at any scale. Fixed by adding the `List-Unsubscribe-Post` header to every send, and giving `SubscriptionConfirm` a third path alongside its existing GET-confirm/POST-act split (§2a): a bare POST whose body is the RFC 8058-mandated literal `List-Unsubscribe=One-Click` is recognized (it can only ever come from a mail client's own automated request, never from prefetch-vulnerable GET scanning) and acted on immediately, with no confirm page — reading action/type/token/uid from the URL's query string, since that's where a header-triggered request puts them rather than in the POST body. The *visible* body unsubscribe link a human clicks is untouched and still gets the §2a confirm-button page.
- **The newsletter subscribe endpoint let anyone probe whether an email address was on the list.** `Subscriber::subscribe()` returned a distinguishable `WP_Error` (HTTP 409) for an already-confirmed address versus a 201 for anything else, so a caller could enumerate subscribers by status code alone. Fixed by making every outcome — new, still-pending, or already-confirmed — return the same success shape and the same 201 response; only whether a confirmation email goes out differs (skipped entirely when already confirmed, since there's nothing left to confirm), and that's invisible to the caller.
- **Pending subscribers and their tokens never expired, and every resubmission resent a confirmation email.** A bot hitting the signup form (still bounded by the existing 5/IP/5min rate limit) could accumulate `pending` rows forever, each triggering its own confirmation email, with no cleanup. Fixed with two independent pieces of hygiene: (1) `Subscriber::expire_stale_pending()` deletes abandoned `pending` rows older than 14 days — piggybacked on the existing daily `agnosis_prepare_newsletters` cron (`Scheduler::prepare()`) rather than a new scheduled event — and never touches `confirmed`/`unsubscribed` rows; (2) resubscribing a still-pending address within a 5-minute cooldown now leaves the row and its token completely untouched (no fresh token, no repeat email) instead of resending every time, while a resubmission past the cooldown still resends normally with a fresh token. Both the age check and the cooldown check are done in SQL against MySQL's own `NOW()` rather than PHP's clock, so there's no timezone-mixing risk of the kind fixed separately below (§3e).
- **Newsletter scheduling mixed three different clocks.** `Scheduler::since_window()`'s never-sent-before fallback used `gmdate()` (true UTC) while every other path returns `sent_at` (a WP site-local wall-clock string, from `current_time('mysql')`) — so on a site with a non-zero UTC offset, the very first digest's content window was shifted against `post_date` (also site-local), and `is_due()`'s bare `strtotime($last_sent)` silently substituted PHP's own ini `date.timezone` for the site's actual configured WordPress timezone when converting back to a Unix epoch, double-converting the offset against `time()`. Fixed by keeping every scheduling timestamp on the one clock the rest of the pipeline (`post_date`, `resolved_at`) already uses — site-local — and using WP core's own `get_gmt_from_date()` (DST-correct, via `wp_timezone()`) for the one place an actual Unix-epoch comparison is needed (`is_due()`), instead of a bare `strtotime()`.
- **Four small hygiene nits.** The newsletter signup block's email `<label>` had no `for`/`id` association with its input, so screen readers had no programmatic link between them — fixed with a matching `id`/`for` pair. `agnosis_newsletter_queue`'s `sent_at` column was renamed to `resolved_at` — it was stamped on a terminal failure just as often as a real send, and the old name was misleading; existing installs get the rename via a guarded `ALTER TABLE`. `agnosis_newsletter_subscribers.email` was shortened from `VARCHAR(255)` to `VARCHAR(191)` — its `UNIQUE` index under `utf8mb4` exceeded the 767-byte limit on older MariaDB/MyISAM configurations; 191 (WordPress core's own long-standing convention for indexed email columns) stays safely under it everywhere. The `SubscriptionConfirm::render_page()` hardcoded-`lang="en"` nit turned out to already be resolved as an incidental side effect of the §2a `wp_die()` conversion earlier in this release — no further change was needed there.

### Fixed
- **A newsletter issue could get stuck in `sending` forever when every recipient's send failed.** `QueueProcessor::process()` previously only re-checked (and possibly marked `sent`) issues that had at least one row transition to `sent` in that batch (`$sent_per_issue`). If every recipient's `wp_mail()` call failed — e.g. a single-artist newsletter hitting a temporary mail delivery problem — none of its rows ever reached `sent`, so the issue was never reconciled; and since its rows were already `failed` (not `pending`), no future cron tick would ever look at it again either. The gate is enforced at `apply()` and re-checked at `maybe_admit()` (a pending application that reaches its vouch threshold while full is parked, not admitted). When a member permanently leaves (`agnosis_artist_left` / `agnosis_removal_vote_passed` / `agnosis_artist_deleted_by_admin`), the oldest waitlisted applicant is advanced back into the vouching flow ("open a slot, fill a slot") and admitted at once if it already had enough vouches. Admins may admit over the cap (logged). Configurable in Settings → Network; the join form reports the full/waitlist state.
  - *Phase 2 (community vote):* a new `CommunityCapVote` governance flow mirroring the removal-vote machinery — any artist proposes a new cap (`POST /agnosis/v1/cap/propose`), it opens as a full community vote once it gathers the configured co-signers (`agnosis_cap_proposal_threshold`, default 3), members vote (`/cap/cosign/{id}`, `/cap/vote/{id}`), and the daily `agnosis_check_cap_votes` cron closes the window: a strict majority (> 50 %) of active artists voting yes writes the new cap and advances the waitlist. New tables `agnosis_cap_proposals` + `agnosis_cap_votes`; `CommunityCapNotification` emails members when a vote opens and the admin on resolution. (audit §8) *(A member-facing voting block + one-click email links — mirroring the removal-vote HMAC links — are a noted follow-up; the governance backend and REST API are complete.)*

- **Token generation** — review and removal tokens now use `bin2hex(random_bytes(32))` (256-bit CSPRNG) instead of `wp_hash()` (HMAC-MD5 over predictable inputs). Extracted into `PostCreator::generate_token()` and covered by `TokenGenerationTest` (format, uniqueness, `hash_equals` compatibility).

- **Post idempotency** — `PostCreator::create_post()` queries `_agnosis_queue_id` meta before inserting; reprocessing a queue row updates the existing post in-place instead of creating a duplicate

### Fixed

- `enqueue()` INSERT silently no-oped when `is_already_queued()` false-returned for existing rows; replaced with `INSERT IGNORE` + follow-up SELECT for true idempotency
- Binary attachment data corrupted by `wp_json_encode()` — function runs UTF-8 sanitisation which destroys JPEG/PNG bytes; attachments are now base64-encoded before JSON storage and decoded in `PostCreator::handle()`
- Empty artwork posts (title only, no body/tags/images) caused by the above corruption silently failing the AI pipeline
- `wp_tempnam()` / `wp_handle_sideload()` undefined in cron and `admin-post` contexts — added `require_once ABSPATH . 'wp-admin/includes/file.php'` before first use
- Settings page asset enqueue hook corrected to `agnosis_page_agnosis-settings` after submenu restructure (was `toplevel_page_agnosis-settings`)
- `wp_get_post_tags()` return typed as `array|WP_Error`; guarded with `is_array()` before passing to `implode()`
