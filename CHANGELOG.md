## [0.2.0] — 2026-06-29

### Added
- **Admission revocation — three independent departure paths (§5h)**

  - **Self-removal** — an artist submits `POST /agnosis/v1/departure/request`. A confirmation email is sent (localised via `switch_to_locale`) with an HMAC-signed link. Clicking the link calls `Departure::confirm_self_removal()`: deletes all `agnosis_artwork`, `agnosis_biography`, and `agnosis_event` posts permanently (bypass trash), removes the `agnosis_artist` role, marks the membership row `status='left'`, and calls `wp_delete_user()`. The admin receives a brief notice; the community is not named.

  - **Admin revocation** — two actions exposed in Settings → Network → Members:
    - **Suspend** — `admin_post_agnosis_ban_artist` sets `status='banned'`, removes the role, and sends a localised suspension email. An optional date picker restricts the ban to a specific day; leaving it blank creates an indefinite ban. A daily cron (`agnosis_check_bans`) automatically reinstates artists whose `banned_until` has passed (role re-granted, status → `admitted`, reinstatement email sent).
    - **Delete** — `admin_post_agnosis_delete_artist` permanently deletes all content and the WP account via `Departure::admin_delete()`. Requires a JS confirmation dialog.

  - **Community vote** — any artist can nominate another via `POST /agnosis/v1/removal/nominate/{id}`. When nominations reach the configurable threshold (default 3, setting `agnosis_removal_nomination_threshold`) **or** an admin triggers `admin_post_agnosis_initiate_removal_vote` directly, the vote opens to the whole community. Each artist receives a personalised email with HMAC-signed yes/no vote links (stateless; tokens verified by `template_redirect`). Artists can change their vote by following the other link before the deadline. A daily cron (`agnosis_check_removal_votes`) resolves expired votes: if >50 % of active members voted yes the artist is removed (same effect as self-removal); otherwise the vote fails and no action is taken. Admin receives an outcome email in both cases.

- **`Departure` class** (`includes/Artist/Departure.php`) — business logic for all three departure paths. Key public API: `request_self_removal()`, `confirm_self_removal()`, `admin_ban()`, `admin_delete()`, `check_expired_bans()`, `nominate()`, `cast_vote()`, `admin_open_removal_vote()`, `check_expired_removal_votes()`, `execute_removal()`, `delete_artist_content()`.

- **`DepartureNotification` class** (`includes/Artist/DepartureNotification.php`) — hooks into actions fired by `Departure` to send all departure-related emails: `agnosis_departure_confirmation_requested`, `agnosis_artist_left`, `agnosis_artist_banned`, `agnosis_artist_reinstated`, `agnosis_removal_vote_opened`, `agnosis_removal_vote_passed`, `agnosis_removal_vote_failed`. Per-recipient locale switching applied throughout.

- **Members dashboard** — Settings → Network now shows an admitted/banned **Members** card below the pending applications table. Each row displays artist name/email, status (Active or Banned with expiry date if applicable), joined date, and action buttons: **Suspend** (with optional expiry date picker), **Delete** (with confirmation dialog), **Open Vote** (admin-bypass community vote). Banned artists only show the Delete button.

- **`agnosis_removal_nomination_threshold` and `agnosis_removal_window_days` settings** (Network tab) — configure how many nominations open a community vote (default 3) and how long the vote stays open (default 7 days).

- **Schema additions** — `agnosis_applications` table extended with `banned_until DATETIME NULL`, `removal_token VARCHAR(64) NULL UNIQUE`. Status ENUM extended to include `banned`. New tables `agnosis_removal_requests` and `agnosis_removal_votes` track community vote state and individual votes. `maybe_upgrade()` guards ensure backward compatibility. Two new daily crons registered on activation: `agnosis_check_bans` and `agnosis_check_removal_votes`.

- **`agnosis/join` block** — dynamic server-side-rendered block that displays the public application form (name, email, bio, portfolio URL, "Why do you want to join?"). Submits via `fetch` to `POST /agnosis/v1/admission/apply`. Shows a success message on acceptance or a server error message on rejection. Admitted artists see a short confirmation instead of the form. Enqueues `frontend.js` and `frontend.css` (no build step — vanilla JS).
- **`/join/` page auto-created on activation** — `Activator::create_join_page()` creates a published page at slug `join` with `<!-- wp:agnosis/join /-->` as content. Idempotent; stored in `agnosis_join_page_id` option.

- **`AdmissionNotification`** — sends personalised emails to admin and every current artist when an application is received, with HMAC-signed yes/no vote links. Artists can change their vote at any time within the voting window. When an application expires without reaching the threshold: applicant is notified the community declined; artists receive a brief closure notice. Newly admitted artists receive a zero-configuration welcome email: password-reset link + a formatted list of all configured alias email addresses (submit, bio, event, replace, remove, promote) so they can start sharing work immediately without any additional documentation. Aliases are only listed when their option is configured — partial setups show only what is available. All email body strings are wrapped in `__()` for `.po`/`.mo` translation. Every outgoing email is now wrapped in `switch_to_locale()` / `restore_current_locale()`: vote emails switch to each artist's WP user locale; expiry and welcome emails switch to the applicant/artist's language.
- **Per-recipient locale switching across all notification emails** — `Publishing\Notification` (`on_post_drafted`, `on_removal_requested`, `on_submission_rejected`) wraps each `wp_mail()` call in `switch_to_locale($artist_locale)` so translated strings are correct for every recipient regardless of the site's default language. In `on_post_drafted`, only the AI-generated `post_excerpt` is back-translated to the artist's language; the artwork title (`post_title`) is intentionally left in the site language — artwork titles are proper nouns and creative decisions by the artist that must not be silently altered. The artist's original submitted title is shown separately in the review email (see `_agnosis_original_title` above).
- **Applicant language capture** — `POST /agnosis/v1/admission/apply` accepts an optional `language` parameter (ISO 639-1 code, e.g. `es`). When omitted, the primary language tag is extracted from the `HTTP_ACCEPT_LANGUAGE` header. The code is stored in `agnosis_applications.language`. On admission, it is mapped to a WP locale (e.g. `es` → `es_ES`) and saved to the user's `locale` meta via `wp_update_user`.
- **Language select on join form** — the `agnosis/join` block now includes a "Language you work in" select field with all 29 languages matching the AI translation map (auto-detect default). The selection is sent as `language` in the application payload.
- **`SubmissionTranslator::translate_text()`** — new public method for back-translating a single string to a given ISO 639-1 target code. Used by `Publishing\Notification::on_post_drafted()` to translate the AI-generated post excerpt back to the artist's language before building the review email. The post title is intentionally not translated — see below.
- **Original artwork title preserved as `_agnosis_original_title` post meta** — the artist's email subject (stripped of any `[Indicator]` prefix, before AI translation rewrites it) is captured in `PostCreator::handle()` and stored once on first post creation. Never overwritten on resends or singleton merges. Used by `Notification::on_post_drafted()` to surface the artist's own words in the review email subject line and as a subtitle beneath the AI-generated site title in the email body.
- **URL slug derived from original submitted title** — `PostCreator::make_slug()` generates the `post_name` from `_agnosis_original_title` rather than the AI-translated `post_title`, so an artist whose subject is "El Jardín Secreto" gets the permalink `…/el-jardin-secreto/` rather than `…/the-secret-garden/`. For non-Latin scripts (CJK, Arabic, Devanagari, Cyrillic, Korean, Thai, etc.) the PHP `intl` extension's `Transliterator` converts to a Latin/ASCII approximation using the ICU rule chain `Any-Latin; Latin-ASCII; Lower()` (e.g. "秘密花園" → `mi-mi-hua-yuan`, "الحديقة السرية" → `alhdy-alsry`). Degrades gracefully on hosts without `intl`: WordPress derives the slug from the translated `post_title` instead.
- **`SubmissionTranslator::from_settings()`** — static factory method that creates a translator from the currently configured AI provider without requiring constructor injection. Returns `null` when no API key is configured so callers degrade gracefully.
- **`VouchConfirm`** — `template_redirect` handler that processes the signed email vote links. Verifies the HMAC token (stateless — no DB write for the token itself), records the vote via `Admission::record_vote()`, and renders a minimal self-contained HTML thank-you or error page without loading the theme.
- **Community voting: yes/no votes, dynamic threshold, 7-day window** — `Admission::vouch()` now accepts a `vote` parameter (`yes` | `no`, default `yes`). Votes are stored with `INSERT … ON DUPLICATE KEY UPDATE` so artists can change their mind. Only positive (`yes`) votes count toward the admission threshold. `calculate_required()` returns `max( ceil( active_artists × percent / 100 ), minimum )` using two new settings (`agnosis_admission_percent` default 10%, `agnosis_admission_minimum` default 3). `check_expired_applications()` runs via daily cron; rejects pending applications older than `agnosis_admission_window_days` (default 7) that have not reached the threshold.
- **Dual-title artwork architecture** — `post_title` is now always the artist's original submitted title (their language, their words — never rewritten by AI). The AI-generated site-language title is stored separately in `_agnosis_translated_title` post meta. `_agnosis_original_title` is retained for backwards compatibility and mirrors `post_title` on first creation. Review emails surface both: the `<h2>` heading shows the original title; a "On the site:" line shows the AI title; a "In your language:" back-translation hint shows the AI title translated back to the artist's locale.
- **`agnosis/artwork-title` block** — server-side-rendered block registered via `Profile::register_blocks()`. Reads `post_title` (original) and `_agnosis_translated_title` meta (AI translation). When the two strings differ, renders an `<hgroup>` with the original as `<h1>` and the translation as a styled `<p>` subtitle. When identical (same-language artist on same-language site), renders a plain `<h1>`. Replaces `wp:post-title` in `single-agnosis_artwork.html`.
- **Applicant acknowledgment email** — when an artist submits an application (`POST /agnosis/v1/admission/apply`), they now immediately receive a confirmation email in their own language via `AdmissionNotification::send_application_acknowledgment()`. The email is localised via `switch_to_locale()` / `restore_current_locale()` using the applicant's submitted language code. Previously applicants received no immediate feedback.
- **HTML email `lang` attribute localised** — all three HTML email templates in `Publishing\Notification` previously hardcoded `<html lang="en">`. A new `html_lang()` helper derives the BCP 47 tag from the current locale (after `switch_to_locale()` has been applied) so the `lang=` attribute matches the actual language of the email content.
- **Language list driven by installed WP language packs** — `JoinPage.php` no longer maintains a hardcoded 29-language array. `SubmissionTranslator::language_names()` (new public static method) reads `get_available_languages()` + the site locale, reduces them to 2-letter ISO codes, and intersects against `LANGUAGE_NAMES` so the join form only offers languages the site actually has localisation files for. Falls back to the full `LANGUAGE_NAMES` constant on a fresh English-only install. Filterable via `agnosis_translation_languages` for operator overrides. No Lingua Forge dependency — plain WP core.
- **Welcome email extended** — the admission welcome email now includes the artist's personal gallery URL (subdomain when `agnosis_base_domain` is configured, falling back to `home_url()` via `SubdomainRouter::url_for_artist()`), a direct link to `/my-submissions/`, and a subject-line convention reference (`[Biography]`, `[Event]`, artwork default). Everything the artist needs in a single email.
- **`agnosis_email_goodbye` alias** — an optional email address (Settings → Email) artists can write to in order to request self-removal without needing access to the website. Sending any email (no attachment required) to this address triggers the same confirmation-email flow as `POST /agnosis/v1/departure/request`: a single-use token link is sent back; clicking it permanently deletes the artist's content and account. Both the IMAP poller (`Inbox`) and the webhook endpoint (`Webhook`) handle the alias; unrecognised or non-artist senders are silently ignored and logged. The goodbye address is listed in the admission welcome email under "To leave the network and delete your account." `Departure::request_self_removal()` refactored to delegate to the new `initiate_removal_for_user(int $user_id): bool` method so both REST and non-REST paths share the same logic.

- **Admin admission dashboard** — Settings → Network now shows a **Pending Applications** card below the configuration fields. The table lists applicant name/email, applied date, a vote progress bar (received/required), bio excerpt, portfolio link, and admin-override **Admit** / **Reject** buttons. Admit (`admin_post_agnosis_admit_application`) bypasses the community threshold: creates the WP user, grants `agnosis_artist`, and fires `agnosis_artist_admitted` so the welcome email is sent. Reject (`admin_post_agnosis_reject_application`) marks the row rejected and fires `agnosis_artist_rejected`. Both actions are nonce-protected, `manage_options`-gated, and redirect back to the Network tab with an inline success or error notice. `Admission::admin_admit()` and `Admission::admin_reject()` are new public methods.

### Fixed
- **LF translation trigger was a silent no-op** — `LinguaForge::request_translations()` was calling `do_action('linguaforge_request_translation', …)`, a hook that does not exist anywhere in Lingua Forge. No translations were ever triggered after artwork publication. Fixed by replacing the `do_action` with direct calls to `linguaforge_trigger_translation(int $post_id, string $target_lang, array $params)` — LF's actual public API in `ai/ai.php` — once per target language, guarded by `function_exists()`.

### Changed
- **Application-first admission flow** — no WP account is created before the community approves. `apply()` is now unauthenticated and stores the application in the new `agnosis_applications` table (email, display_name, bio, portfolio_url, statement). Artists vouch against the application ID via `agnosis_application_vouches`. `maybe_admit()` calls `wp_create_user()` only when the vouch threshold is reached, then grants `agnosis_artist` and fires `agnosis_artist_admitted`. Reapplication is allowed for withdrawn/rejected/left entries by updating the existing row.
- **Membership lifecycle table (`agnosis_applications`)** — tracks every applicant and member with statuses `pending → admitted | rejected | withdrawn | left`. `resolved_at` timestamps every transition. `wp_user_id` is null until admission. Serves as the authoritative membership status table for future leave-community and re-join flows.
- **Pre-admission vouch table (`agnosis_application_vouches`)** — separate from `agnosis_vouches` (post-admission community table). Revocation sets `revoked_at` rather than deleting so the audit trail is preserved.
- **Status endpoint is admin-only** — `GET /agnosis/v1/admission/status/{application_id}` now takes an application ID and requires `manage_options`. Applicants are informed of state changes by email only.
- **Self-vouch guard by email** — previously compared WP user IDs; now compares the vouching artist's `user_email` against the application email, correctly covering the case where the applicant has no WP account yet.

## [0.1.9] — 2026-06-28

### Added
- **`/my-submissions` works on artist subdomains** — `SubdomainRouter::scope_query()` previously applied the `author` filter to every main query, including static page requests. Since the `/my-submissions` page is created by the system (not the artist user) it returned a 404 on any artist subdomain. A guard now skips the author and post_type filters when `pagename` is set or the query already targets `post_type=page`, letting WordPress resolve sitewide pages normally. Session cookies must also carry across subdomains — add `define('COOKIE_DOMAIN', '.agnosis.art');` to `wp-config.php` so artists logged in on the main domain are also authenticated on their subdomain.
- **Inline login form on `/my-submissions`** — unauthenticated visitors no longer see a plain "Please log in" notice or get bounced to `wp-login.php`. `SubmissionsPage` now renders an inline form via `wp_login_form()` with the current page URL as the redirect, so the artist logs in and lands back on their submissions page without ever leaving the frontend.
- **Artist name in subdomain site title** — `SubdomainRouter` now filters `option_blogname` on artist subdomains so the `wp:site-title` block (and anything else reading `get_bloginfo('name')`) displays the artist's display name instead of the main portal site name. Falls back to the nicename slug so the header is never blank. The existing `document_title_parts` and `wp_title` filters (HTML `<title>`) already did this; `option_blogname` closes the gap for the visible header.
- **Multilingual submission translation** — artist emails are automatically translated to the site's primary language before AI processing. The target language is resolved from the Lingua Forge `linguaforge_primary_language` option, with fallback to the WordPress site locale and a final fallback to English. Translation is performed in a single `chat()` call (one round-trip) using a JSON envelope that carries both the email subject and body. If translation fails (empty or non-JSON response) the original text is passed through unchanged — the pipeline always continues. The target language is extensible via the `agnosis_translation_languages` filter. Translation events are logged to Settings → Logs (`pipeline` context). Implemented in new `SubmissionTranslator` class (`includes/AI/SubmissionTranslator.php`); `chat()` max_tokens bumped from 512 → 1024 in OpenAI and Anthropic to accommodate translated email bodies.
- **Media type expansion** — the AI pipeline now handles audio, video, and PDF attachments in addition to images:
  - **PDF portfolios** — each page is rasterised to a JPEG at 150 dpi using the PHP Imagick extension and fed into the normal describe → optionally-enhance image branch. Multi-page PDFs produce one gallery image per page. Falls back gracefully (skips with a logged notice) when Imagick is not installed.
  - **Video stills** — the first frame is extracted via `ffmpeg` and processed as a JPEG through the image branch. Enhancement is applied normally. Falls back gracefully when `ffmpeg` is not in `PATH`.
  - **Audio files** — routed through a new audio branch: (1) transcribed via OpenAI Whisper (`POST /v1/audio/transcriptions`, `whisper-1`); (2) the transcript + artist context is sent to the text model (`chat()`) with a structured JSON prompt to produce `title`, `excerpt`, `body`, `tags`, `alt_text`, and `medium`; (3) no image is uploaded to the Media Library — `PostCreator::merge_gallery()` skips `upload_image()` for `media_type === 'audio'` results. Providers without audio support (`Anthropic`, `WordPressAI`) return an empty transcript and fall back to describing from artist text alone.
  - **`MediaAdapter`** — new `includes/AI/MediaAdapter.php` class with a single static `adapt(array $attachments): array` method. Sits in front of the pipeline loop; expands each attachment into zero or more normalised entries with a `media_type` field (`'image'` or `'audio'`). Unsupported MIME types are logged and dropped.
  - **`ProviderInterface`** extended with `transcribe(string $audio_data, string $mime_type): string` and `supports_audio(): bool`; existing providers updated.
- **Automatic quality rejection gate** — when the primary image's quality score is at or below the configurable rejection threshold (default: 3 / 10), the submission is rejected automatically: the queue row is marked `failed`, no draft post is created, and the artist receives a friendly HTML email explaining the specific photographic issue and how to fix it. Issue labels from the vision AI (e.g. "too dark", "motion blur", "glare") are mapped to plain-language advice with concrete tips ("Try photographing near a large window in daylight…"). The email also includes four universal photography tips. Artists are invited to resubmit — the entire flow is automated with no admin action required. Configurable via Settings → AI Providers: **Rejection threshold** (set to 0 to disable). - **`Pipeline::extract_event_fields()`** — lightweight chat pass that runs after the polish step for `agnosis_event` submissions; sends body + subject to the text model and returns a JSON object with a `location` key; strips markdown fences from the response; short-circuits (returns `['location' => '']`) when body and subject are both blank. No admin entry required — location is fully automated.
- **`_agnosis_event_location` post meta** — stored on every `agnosis_event` post via `meta_input` during `PostCreator::create_post()`; populated from `Pipeline::extract_event_fields()` and falls back to an empty string when the AI finds no location in the email.
- **`agnosis/event-location` block** — server-side-rendered block registered in `Profile::register_blocks()`; uses `postId` context from the query loop; reads `_agnosis_event_location` meta and renders a styled `<p class="agnosis-event-location">` (small, bold, uppercase, letter-spaced); returns an empty string when meta is unset so events without a location leave no visual gap.
- **`agnosis_email_promote` setting** (Email tab) — configurable promote@ address (e.g. `promote@agnosis.art`); description explains the workflow: artist sends an email with the exact artwork title as subject to explicitly feature that artwork in their gallery overview.
- **`PostCreator::handle_promotion_request()`** — handles the `agnosis_promote` pseudo-type; finds the sending artist's published artwork by exact title match; calls `set_featured()` if found; logs a warning and exits silently when subject is empty or no match is found.
- **`PostCreator::set_featured()`** — sets `_agnosis_featured = '1'` on the target artwork and demotes all other published artworks by the same artist to `'0'`; drafts and other artists are not affected. Kept in the publishing layer (not `GalleryOverview`, which is a display class) because featuring is a workflow decision.

### Fixed
- **`CREATE TABLE IF NOT EXISTS` removed from `dbDelta()` calls** — `dbDelta()` handles the "table already exists" case itself by diffing the schema against the current table; the `IF NOT EXISTS` clause caused it to short-circuit before performing that comparison, silently skipping new columns on existing tables. All five `CREATE TABLE IF NOT EXISTS` statements in `Activator::create_tables()` are now plain `CREATE TABLE`. The pre-existing manual `ALTER TABLE` guards in `maybe_upgrade()` (`post_id`, `revoked_at`) are retained for backward compatibility with installs created before this fix.

### Changed
- **Auto-promotion on approval removed** — `ReviewEndpoints::approve()` and `save(publish=true)` no longer set `_agnosis_featured`; the entire `promote_featured()` method has been deleted. Featuring is now always explicit — the artist sends an email to the promote@ alias with the artwork title as subject.
- **Theme 0.1.5** — `single-agnosis_biography.html` rebuilt with a two-column layout (left 38%: featured image at 3/4 aspect, 4 px border-radius; right 62%: title + spacer + content; columns stack on mobile; separator + author/year footer preserved); `single-agnosis_event.html` updated to group the post date and `wp:agnosis/event-location` block together with reduced block gap.

### Fixed
- **`ReviewEndpoints::reject()` incorrect action arg count** — `do_action('agnosis_submission_rejected', $post_id)` was passing only 1 argument; hooks registered with `accepted_args=4` received `$post_id` in the `$score` parameter position instead of `$author_id`. Fixed to pass all four expected args: `$post_id, (int) $post->post_author, 0, []`.
- **`render.php` unescaped `$agnosis_thumb_id`** — attachment ID in `class="wp-image-{id}"` was emitted raw; wrapped with `absint()`.
- **Minimum WordPress version bumped to 6.6** — `wp_interactivity_state()` and `wp_enqueue_script_module()` were both introduced in WP 6.5; declaring 6.4 as minimum caused Plugin Check errors. Updated plugin header, `AGNOSIS_MIN_WP` constant, and `readme.txt`.

## [0.1.8] — 2026-06-28

### Security
- **Token generation** — review and removal tokens now use `bin2hex(random_bytes(32))` (256-bit CSPRNG) instead of `wp_hash()` (HMAC-MD5 over predictable inputs). Tokens cannot be reconstructed from artist ID, timestamp, or site secrets. Extracted into `PostCreator::generate_token()` and covered by `TokenGenerationTest` (format, uniqueness, `hash_equals` compatibility).
- **REST endpoint permission gates** — `ReviewEndpoints` (`PUT /review/{id}`, `POST /review/{id}/approve`, `POST /review/{id}/reject`) and `RemovalEndpoints` (`POST /removal/{id}/confirm`) previously used `'permission_callback' => '__return_true'`, relying entirely on the internal `check_access()` / `verify_token()` logic. Added `check_permission()` to each class as an explicit outer gate: returns `true` when a token param is present or the user is logged in, `WP_Error(401)` otherwise.
- **Token URL shim** — email action links (approve, reject, remove) previously linked directly to REST endpoints with `?token=` in the query string; tokens appeared in server access logs, browser history, and Referer headers. Added `ReviewConfirm` class, registered on `template_redirect`: email links now point to `/?agnosis_review=1&id={id}&action={action}&token={token}`. The shim calls `rest_do_request()` with the token in the POST body (no logged REST URL) and immediately redirects to `/?agnosis_result={action}` — a clean URL. Browser history and Referer headers never contain the token.
- **ActivityPub inbox HTTP Signature verification** — `POST /activitypub/inbox` previously accepted any payload from any IP. Added `HttpSignature` class implementing Fediverse-compatible HTTP Signature verification: parses `keyId`/`headers`/`signature` from the `Signature` header; enforces a ±12 h Date freshness window; verifies the `Digest: SHA-256=…` body hash when present; fetches `publicKey.publicKeyPem` from the sender's actor document (cached 24 h per keyId); reconstructs the signing string and verifies RSA-SHA256 via `openssl_verify()`. Wired as `verify_inbox_signature()` permission callback on the inbox route — a spoofed or unsigned Follow/Announce is rejected with 401/403 before `inbox()` runs.

### Security (continued)
- **Admission status endpoint now requires authentication** — `GET /agnosis/v1/admission/status/{id}` previously returned `is_artist`, `has_applied`, and vouch counts for any WP user ID without authentication, allowing enumeration of the entire artist/applicant roster. Added `require_logged_in_for_status()` permission callback (returns 401 for anonymous requests) and an ownership guard inside `status()`: a non-admin user requesting a different user's status receives 403. Administrators retain full access. Covered by `AdmissionStatusTest` (7 assertions: 401 anonymous, 403 cross-user, 200 own data, field correctness, admin access to any user, admin field accuracy, artist own status).
- **Per-IP rate limiting on webhook and REST endpoints** — new `Core\RateLimiter` class uses WordPress transients to count requests per action per IP within a rolling time window. The window slot rotates automatically via `floor(time() / $window)` so counters expire without a cleanup cron. Limits: `POST /admission/apply` → 5 req/min (wired in `Admission::require_logged_in()`); `POST /admission/vouch/{id}` → 20 req/min (in `Admission::require_artist()`); `POST /email/inbound` → 60 req/min (in `Webhook::verify_signature()`). Exceeding the limit returns `WP_Error('agnosis_rate_limit', …, ['status' => 429])`. IP detection uses `REMOTE_ADDR` only — not `X-Forwarded-For` — to prevent header spoofing; sites behind a trusted reverse proxy that rewrites `REMOTE_ADDR` (nginx `real_ip_from`, AWS ALB, Cloudflare) work correctly without configuration.

### Changed
- **`WP_Filesystem` used for all filesystem writes** — replaced bare `file_put_contents()` / `file_get_contents()` calls in `PostCreator::upload_image()` and `Email\AttachmentStore` with `WP_Filesystem` (`'direct'` method forced via filter, since FTP/SSH credentials cannot be prompted in cron or admin-post context). Extracted as a private `filesystem()` helper on `PostCreator` and an equivalent static helper on `AttachmentStore`.
- **Attachment binaries moved out of the database** — `Inbox::enqueue()` now writes each image binary to `uploads/agnosis-queue/{uid}/{index}-{filename}` via the new `Email\AttachmentStore` class; `raw_email` JSON stores a `file` path instead of a base64 blob, eliminating 5–10 MB DB payloads per submission. `PostCreator` reads the file and deletes it after a successful media library upload. `AttachmentStore::sweep_orphans()` is called by the daily cleanup cron to remove dirs left by failed rows. - **Vouch revocation** — added `revoked_at DATETIME NULL DEFAULT NULL` to `agnosis_vouches`; existing installations receive the column via `maybe_upgrade()` on the next page load. `count_vouches()` filters `AND revoked_at IS NULL`. Covered by `VoucheRevocationTest`.
- **`handle_test_ai()` duplicate HTTP logic extracted** — the identical `wp_remote_post` + error-handling block that was copy-pasted for OpenAI and Anthropic is replaced by a shared private `Settings::ping_provider()` method. Adding a fourth provider now costs 5 lines, not 18.

Called at the end of every `poll()` run regardless of IMAP configuration. Covers webhook-sourced rows whose initial single event was missed due to no page load after ingest, and sites running in webhook-only mode. `wp_schedule_single_event` deduplicates identical events already queued so this is safe to call every 5 minutes. Covered by `QueueDrainTest`.

- **`upload_size_limit` filter cleanup scoped to own closure** — `PostCreator::upload_image()` was calling `remove_all_filters('upload_size_limit')` after `wp_handle_sideload()`, silently removing any filters added by co-active plugins. - **`no_found_rows` on all ID-only `get_posts()` calls** — added `'no_found_rows' => true` to 8 calls across `PostCreator` (5), `Admin\InboxPage` (2), and `Network\ActivityPub` (1) that fetch IDs and never read `$query->found_posts`. Production code passes nothing; tests inject a stub that skips provider resolution and returns canned results without hitting AI endpoints. Covered by `PipelineInjectionTest`.

New `Admin\QueueController` takes the five queue admin-post handlers (`poll_now`, `force_reprocess`, `process_queue`, `process_one`, `delete_queue_row`); `handle_test_inbox` moved to `Admin\InboxPage`; `handle_test_ai` and `handle_clear_logs` moved to `Admin\Settings`; `compatibility_notices()` moved to `Compat\LinguaForge`; `ensure_roles()` extracted to new `Core\Roles`; `register_image_sizes()` extracted to new `Core\ImageSizes`. `Plugin.php` is now 145 lines of hook wiring only.
