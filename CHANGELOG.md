# Changelog

All notable changes to Agnosis are documented here.

Format: [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) —

## [0.8.2] — 2026-07-06

### Fixed
- **A second, independent cause of the "Sending…" freeze fixed in 0.8.1, found while verifying that fix on a live site: the `agnosis_send_newsletter_queue` cron event itself was not scheduled at all** (`wp cron event run agnosis_send_newsletter_queue` failed with "Invalid cron event") — so the queue had never been processed even once, let alone reconciled. The only code path that registers this event, `schedule_events()`, previously ran only from `activate()` or a version-bump-gated `maybe_upgrade()` — so if it's ever lost for a reason unrelated to a version bump (a host's cron-table cleanup, a manual `wp cron event delete`, a migration between servers), nothing would notice or recover until the next plugin update. New `Activator::ensure_newsletter_cron_scheduled()` is called unconditionally from the Newsletter dashboard on every page view, so a missing cron event self-heals the same way a stuck status now does.

## [0.8.1] — 2026-07-06

### Fixed
- **A newsletter issue could get stuck showing "Sending…" (Send Now disabled) forever, even well past the next scheduled cron tick.** `QueueProcessor::process()`'s per-recipient send loop had no exception handling — if `send_one()` threw for any single recipient (a bad locale render, a branding/image error building that particular email, etc.), the uncaught error killed the entire cron tick before reaching `reconcile_sending_issues()`, which runs once per tick after the loop. That meant *every* currently-"sending" issue, not just the one that errored, silently stopped being reconciled — indefinitely, since the exact same failure recurred on every subsequent tick before ever reaching the reconcile step. `send_one()` is now wrapped in a try/catch per recipient: a failure is logged and that row is retried/eventually marked terminally 'failed' exactly as a normal `wp_mail()` failure already was, and reconciliation always runs regardless of what happened to any individual recipient.
- **The Newsletter Status dashboard (Settings → Newsletter) now self-heals a stuck "Sending…" status the moment an admin views the page**, instead of only reflecting reality on WP-Cron's next tick. On a low-traffic install with no real system cron wired to `wp-cron.php`, that tick may not happen again for a long time. `QueueProcessor::reconcile_sending_issues()` is now public and called at the top of `render_newsletter_dashboard()` — cheap (a few `COUNT` queries), a no-op when nothing is actually "sending".

## [0.8.0] — 2026-07-06

### Added
- **Front-end correction for artists — Phase 1 of the third audit's §7 feasibility evaluation.** Admitted artists can now correct post content on the three CPTs, the artwork excerpt, and an event's location/date directly on the published page — no intake email, no dashboard, no forms. New `Artist\ContentEditor`: `POST /agnosis/v1/content/{id}/text` (`{ field, value }`). Authorization mirrors `ReviewEndpoints`'s existing pattern: logged in, `Admission::is_admitted_artist()`, `post_author === get_current_user_id()` (admins bypass via `manage_options`), post type in an allowlist, status `publish`. A conditionally-enqueued overlay script (`blocks/content-editor/frontend.js`) appears only on a singular page the current viewer may edit — decorating `[data-agnosis-edit-field]` regions (written by new `the_content`/`the_excerpt` filters and by `Profile`'s event render callbacks) with a pencil affordance, `contentEditable`, and a floating Save/Cancel pair.
- **Front-end correction is restricted to the artist's own declared language — never a free language switcher** (audit §7c, reassessed 2026-07-06 after re-checking Lingua Forge's actual translation direction — see that day's audit update for the full reasoning). An artist may only edit the post version whose `_lf_lang` matches their own WP user `locale` (already captured at admission, `Admission::apply()`/`iso_to_wp_locale()` — no new field needed). On save: the edit is persisted verbatim to that post; if it isn't the primary-language post, the edit is translated into the primary language via Agnosis's own `AI\SubmissionTranslator` (the same tool `build_title_translations()` already uses for primary → other title translations — deliberately not Lingua Forge's `linguaforge_trigger_translation()`/`ChunkTranslation`, which operate on the whole post and would re-translate fields the artist didn't touch) and written directly to the linked primary post (found via `linguaforge_get_translations()`); the primary post is then fanned out to every other configured language via the existing debounced dispatch, **excluding the artist's own source language**, which already holds their exact words and must not be re-derived by a double round-trip. `Compat\LinguaForge` gained a static `schedule_fanout( $post_id, $exclude_langs = [] )` (self-contained — doesn't require the single live `LinguaForge` instance) and `locale_to_lang()` is now public.
- **Biography edits also update `_agnosis_artist_prompt`** (using the primary-language text when the edit originated in a non-primary language), so the next emailed bio update AI-merges against the corrected text instead of resurrecting whatever the correction just fixed.
- **`'revisions'` added to the `agnosis_artwork`, `agnosis_biography`, and `agnosis_event` CPTs' `supports` array** — every front-end correction becomes a WordPress revision automatically (via `wp_update_post()`), giving admins free undo and an audit trail with no extra code (audit §7c "safety rails").
- Corrections are rate-limited to 30 saves per artist per hour (`RateLimiter::check_sender()`, reused rather than reinvented) and logged via `Logger`.
- **Front-end photo substitution — Phase 2 of the same feasibility evaluation.** Admitted artists can replace the featured image (biography/event) or a specific gallery image (artwork) with a direct upload: `POST /agnosis/v1/content/{id}/photo`, multipart `{ file, attachment_id? }`. Reuses the exact same authorization as Phase 1 (`ContentEditor::check_access()` — own post, own language, admin bypass, rate limit) and the exact same media-library conventions the AI pipeline already uses: `Publishing\PostCreator::upload_media()` (now public) runs the upload through the real `wp_handle_sideload()`/MIME-sniffing path, no new upload code duplicated. Deliberately skips the AI pipeline entirely — no enhancement, no quality gate; a direct replacement is the artist's deliberate choice, the same `photo_only` semantics `PostCreator` already encodes for `photo@` submissions.
- **The replaced photo is never deleted.** It's marked as the new attachment's `_agnosis_original_attachment_id` and flagged `_agnosis_is_original` — the identical provenance-meta pair the AI-enhancement path already uses for its pre-enhancement sidecar — so it stays recoverable for a future "restore original photo" one-click (Phase 3).
- **Photo replacement propagates to every language version of the post synchronously, with no AI call and no deferred dispatch.** Images are language-neutral (audit §7c), so `_thumbnail_id`/`_agnosis_gallery_ids` are copied directly to every post in the same `linguaforge_get_translations()` group the moment the upload completes, rather than going through the text-edit translate-then-fan-out pipeline.
- The overlay decorates the featured image on all three CPTs via a new `post_thumbnail_html` filter (covers both classic and FSE/core `post-featured-image`-block themes uniformly); clicking the pencil opens a native file picker and uploads immediately. Artwork gallery images beyond the featured one are reachable via the same endpoint's `attachment_id` param, but the overlay doesn't yet decorate them individually — left as a fast-follow.
- **Artwork title editing with dual-title handling — Phase 3 of the same feasibility evaluation.** `title` (→ `post_title`) added to the editable-field allowlist for artwork and biography. Artwork's `post_title` is the artist's own words and, by design (`Compat\LinguaForge::hold_artist_title()`), must stay byte-identical on every language version — never translated, the same as a photo. So a title edit runs through a new `ContentEditor::propagate_title()` instead of the normal translate-then-fan-out path: the literal new title is copied verbatim and synchronously to every sibling post in the `linguaforge_get_translations()` group (no AI call, exactly like the photo swap), while the *separate* AI-translated display title (`_agnosis_translated_title`, the subtitle `agnosis/artwork-title` renders) is regenerated independently — translated into the primary language via `AI\SubmissionTranslator` and stored on the primary post, then the *existing* deferred dispatch (`LinguaForge::schedule_fanout()`, unchanged) rebuilds `_agnosis_title_i18n` and refreshes each sibling's own display title the next time it's (re)translated — no new Lingua Forge hook needed. Biography's title has no dual-title system (LF already translates it normally) and reuses the Phase 1 content-style propagation unchanged. `Profile::render_artwork_title()` and a new `the_title` filter (`ContentEditor::decorate_title()`, scoped to the singular main-query post) expose the pencil affordance on the artist's own title; empty titles are rejected (400).
- **"Restore original photo" one-click — the other half of Phase 3.** `POST /agnosis/v1/content/{id}/photo/restore` swaps the current attachment back to whichever one it recorded as its `_agnosis_original_attachment_id` when it was replaced (Phase 2) — no upload, a pure re-pointing via the existing `swap_photo_everywhere()`. The provenance pointer is reversed rather than deleted, so a second restore call undoes the first: nothing Phase 2 already preserved is ever thrown away by this action either. The overlay shows a "Restore earlier photo" action on any photo with a recorded original.
- An optional, explicit "refresh the AI description for the new photo?" action (audit §7c — never automatic) remains deferred: it would need to run through `AI\Pipeline::process()`'s full submission shape (subject/body translation, MediaAdapter routing), which is a heavier integration than a same-cycle fast-follow warranted. Decorating individual artwork gallery images beyond the featured one (the REST endpoints already accept any `attachment_id` in the gallery) is also still a fast-follow, not part of this release.
- **Newsletter archive / "view in browser" — third audit's §8 roadmap item.** Every sent public issue already stores its fully-rendered HTML in `agnosis_newsletter_issues`, so a public archive was nearly free to add. New `Newsletter\Archive` registers two front-end routes with no new CPT/table: `GET /newsletter/` (paginated index, newest first) and `GET /newsletter/{id}/` (a single issue's permalink). Both are `newsletter_type = 'public' AND status = 'sent'` only — an artist-type issue (open community votes, new-member names; see `Digest::build_artist()`) 404s exactly like a nonexistent one, the same as any other issue ID that doesn't match that filter. The single-issue view best-effort matches the visitor's current `get_locale()` against the issue's per-locale `locale_content` map, falling back to the issue's base (default-locale) render — the same fallback `QueueProcessor` already uses for a recipient. Both routes render via `wp_die()`, not a raw echo+exit, matching the convention every other "render a page and stop" flow in this plugin already follows (`SubscriptionConfirm`/`VouchConfirm`/`ReviewConfirm`) — chosen specifically so Archive stays testable via the `wp_die_handler` filter.
- `Mailer::build_email()`'s branded card markup was split into a new `Mailer::build_body()` fragment (no doctype/head/body) so Archive's single-issue page can reuse the exact same visual design while still going through `wp_die()` for its own document wrapper. Every sent email also gained a small "Having trouble viewing this email? View it online." banner linking to `Archive::issue_permalink()` — public issues only, never artist ones — and the footer now shows "Subscribe to get these by email" instead of a broken "Unsubscribe" link when there's no recipient to unsubscribe (the archive page reusing the same markup with `$unsubscribe_url = null`).
- The newsletter signup block now shows a "See past issues" link to the archive once `Scheduler::last_sent_at('public')` confirms at least one issue has ever gone out.
- Existing installs pick up the new `/newsletter/` rewrite rules automatically: `Activator::maybe_upgrade()` now defers a one-time `flush_rewrite_rules()` to `init` (same pattern already used there for `create_managed_pages()`), so no manual permalink resave is needed after upgrading.
- **Locale coverage metric — third audit's §8 roadmap item.** The Newsletter dashboard's Public row now shows a "By language" breakdown of confirmed subscribers (e.g. "Spanish (12), French (5), Unknown (2)") — a cheap signal for which Lingua Forge languages actually earn their AI translation spend, now that §3c (the signup block's missing `language` param) has been fixed since 0.4.3 and `locale` is reliably captured at signup. New `Subscriber::counts_by_locale()` (confirmed-only, `GROUP BY locale`) backs it; `Admin\Settings::locale_label()` resolves a WP locale to a human-readable name via Lingua Forge's own `linguaforge_language_label()` when LF is active (guarded by `function_exists()`, same convention as the rest of `Compat\LinguaForge`), falling back to the raw locale code otherwise. A subscriber with no recorded locale is labelled "Unknown" rather than dropped, so the breakdown always sums to the existing confirmed-subscriber total.
- **Newsletter signup gained the same language selector the join form already has.** The `agnosis/newsletter-signup` block now renders a `<select name="language">` sourced from the same `AI\SubmissionTranslator::language_names()` list `Artist\JoinPage` uses, giving visitors an explicit way to pick their language instead of the block's previous entirely-invisible auto-detect. Deliberately looser than the join form's version: not `required`, no disabled placeholder — `frontend.js` pre-selects whatever `document.documentElement.lang` reports for the current page (the same signal the old auto-detect used) and the visitor can change it before submitting. A wrong or blank choice here is low-stakes and self-correcting (worst case, a digest renders in the site's default-locale fallback until noticed), unlike the join form's language choice, which an artist must confirm explicitly since Lingua Forge may not actually support it. `Newsletter\Subscription`'s REST endpoint gained the same defensive whitelist re-check `Admission::apply()` already has for its own language param — a submitted code outside `language_names()` is silently dropped rather than trusted.
- **Vision-input images are now downscaled before being sent to the AI for description**, a pure cost/latency optimisation: vision-API token cost scales with resolution, and describing a piece of art needs nowhere near the pixel count of the actual published photo. New setting **AI → "Vision image max width (px)"** (`agnosis_ai_vision_max_width_px`, default 800, set to 0 to disable) controls the target width; new `MediaAdapter::maybe_downscale_for_vision()` resizes proportionally with Imagick (never upscales; falls back to sending the original image unmodified if Imagick is unavailable, the source is already narrower than the configured max, or the image data is unreadable) and is called from `Anthropic::describe()` and `OpenAI::describe()` immediately before base64-encoding the request body. Deliberately **not** wired into `MediaAdapter::adapt()`/`adapt_pdf()`/`adapt_video()` themselves — the attachment data those methods return is reused downstream for AI image enhancement and as the actual published artwork/video-poster file, so downscaling it there would have silently shrunk published images instead of just the copy sent for description. Covers all three image-to-vision routes uniformly (plain image attachments, rasterised PDF pages, extracted video poster frames), since all three already converge on `ProviderInterface::describe()`.
- **Artist-facing emails now mention front-end correction (§7's "your work, your corner" idea, scoped down from the original newsletter-personalization proposal in the audit's §8 roadmap once we established there's no dedicated per-artist newsletter to hang it on — see that day's audit discussion).** New `EmailFooter::edit_reminder_plain_text( $artist_id )` / `edit_reminder_html( $artist_id )` sit alongside the existing work-address summary in the same shared class, but are gated per-recipient rather than shown unconditionally: a private `has_published_work()` checks `count_user_posts()` across all three CPTs and returns `''` until that specific artist has at least one published artwork, biography, or event, since the pencil-edit overlay (Phase 1, above) has nothing to decorate before then. Wired into `Notification`'s pre-publish review, removal-confirmation, and photo-quality-rejected emails (artist_id threaded through each `build_*_email()` for the gate), `AdmissionNotification`'s per-artist admission-vote email, `DepartureNotification`'s reinstatement and removal-vote emails, and `CommunityCapNotification`'s vote-opened email. Deliberately **not** added to the suspension email — `Departure::apply_ban()` actually removes the `agnosis_artist` role for the ban's duration, which is the same capability `ContentEditor::check_access()` requires, so the reminder would be false for exactly as long as that email applies; it reappears in the reinstatement email once access is restored. Also skipped: the welcome/admission email and pre-admission applicant emails (nothing published yet, almost by definition) and the BCC'd community-expiry notice (one shared body across many recipients, no single artist to gate on).
- **Password recovery for artists — gap found while reviewing the login/admission flow.** `SubmissionsPage`'s themed inline login form (`wp_login_form()`) never included a "Lost your password?" link — a WordPress core omission for that function, unlike the full `wp-login.php` page — and it was the *only* login surface anywhere in the plugin or theme, so an artist who forgot their password had no discoverable way back in short of already knowing the unlinked `wp-login.php?action=lostpassword` URL exists. Added a "Forgot your password?" link (`wp_lostpassword_url()`) beneath the inline form. Also added new `Core\LoginBranding`, hooked alongside `FrontendAccess`: since "Lost your password?"/"Reset password" necessarily leave the themed front-end page and land on stock `wp-login.php`, it replaces the WordPress logo/link with the site's own (reusing the configured email logo, `agnosis_email_logo_id`, when set) and restyles the submit button and links to match the rest of the site rather than leaving the one unbranded screen in the whole flow.
- **"Invite an Artist" — a new, deliberately simple way to invite a prospective artist by email, independent of the newsletter system.** New `Artist\Invitation`, a small card on Settings → Newsletter below the existing Newsletter Status dashboard: an admin enters one email address, picks a language, and sends immediately — no list, no queue, no recipient tracking (mirrors `Scheduler::send_test()`'s "fire and forget" shape rather than the newsletter's DB-backed audience/queue). Includes its own "Send Test" (identical content, `[TEST]`-prefixed subject, same convention as the newsletter dashboard's test sends). The email body is a standing, admin-editable intro (new **Invitation intro** setting, pre-populated with starter copy describing what Agnosis is and how artists work with it — unlike the newsletter's one-shot intro fields, this one isn't cleared after sending) followed by a fixed "Apply to join" button linking to the real `/join/` page (resolved from `agnosis_join_page_id`, the page `Activator::create_join_page()` recorded, with a same-slug fallback). Language selection works like the newsletter's own admin-intro localisation: the chosen language switches the template's own locale (`switch_to_locale()`, picking up a matching `.mo` when one exists) and the free-text intro itself is machine-translated via the site's configured AI provider when the target isn't the site's source language — falling back to the untranslated original when no provider is configured, the same graceful degradation used everywhere else this pattern appears.

## [0.7.3] — 2026-07-05

### Fixed
- **Artist-subdomain links dropped the visitor's current language.** `SubdomainRouter::url_for_artist()` and `SubdomainNavigation::portal_home_url()` always built bare source-language URLs (`https://slug.base`), regardless of the page the visitor was actually on. On a translated page (e.g. `ourartist.agnosis.art/fr/...`), this meant the artist-name breadcrumb link and the "back to Agnosis" Site Logo/Site Title link both dropped the visitor back to the English root instead of staying on `/fr/`. Added `Compat\LinguaForge::current_lang_path_prefix()` — resolves the current request's language from LF's `LF_LANG` constant (via `linguaforge_source_language()` for the source-language comparison) and returns a joinable `/xx` path segment, or `''` when Lingua Forge isn't active or the current language already is the source language. Wired into both URL builders; as a side effect this also fixes the same latent bug in `gallery-overview/render.php`'s artist links, since they share `url_for_artist()`.

## [0.7.2] — 2026-07-05

### Fixed
- **Gallery-overview block's pagination could disagree with what it actually rendered, on sites using "Your latest posts" as the homepage.** The block built its artist/artwork pool via raw SQL and per-artist `get_posts()` calls with no language filtering of its own, computing `$agnosis_max_pages` from that unfiltered pool — while Lingua Forge's `handle_secondary_pre_get_posts()` hook silently language-filtered only the final by-ID fetch. Translated posts counted toward the page total but never rendered. Fixed by explicitly filtering all three `get_posts()` calls (featured artwork, extra/remaining, final by-ID fetch) in `blocks/gallery-overview/render.php` by `_lf_lang` against the current request's `LF_LANG` constant, gated on `Compat\LinguaForge::is_active()`, so pagination math and rendered content are now always computed from the same pool. The distinct-authors discovery query is deliberately left unfiltered by language — it only determines which artists have any published artwork at all, so no artist is dropped from a language's pool for lack of translations yet.

## [0.7.1] — 2026-07-05

### Fixed
- **Email logo could render far smaller than configured.** `Core\EmailBranding::header_html()` requested the logo at WordPress's `'large'` size — a 1024×1024 *bounding box* WordPress never upscales past. For a wide banner-style logo, the generated `'large'` file could already be shorter than the configured display height before the `max-height` CSS cap even applied, silently undercutting it. Switched to `'full'`, the attachment's actual native resolution, so the cap always has real headroom to work from.
- **Newsletter subscription-confirmation email (`Newsletter\Subscription.php`) predated `EmailBranding` entirely.** It was missed when the shared email-branding system landed in 0.6.4 and still hardcoded its own plain-text "✦ Site Name" header, old purple color, old padding. Now calls `EmailBranding::header_html()` like every other outgoing email and matches the rest of this changelog entry's changes.

### Changed
- **Email logo height raised to 150px** (was 40px) so a full banner-style logo (wordmark + tagline baked into the image, not just an icon) actually fits, instead of being compressed to the point of illegibility.
- **Email header background changed from purple to the same dark colour used on the website** (`#0d0d12`, the theme's own "Background" palette colour) — across all five outgoing HTML email templates (submission review, removal confirmation, rejection notice, subscription confirmation, newsletter digest). Buttons, links, and border accents elsewhere in these emails are unaffected — only the header bar itself changed, via a new `$header_bg` variable kept separate from the existing `$accent` used for interactive elements.
- **Logo now sits on its own white panel inside the header**, fixed at 552px — the same width as the email body content below (600px card − 24px padding each side) — so it lines up edge-to-edge with the content beneath it. Needed because the newly-created logo file has a white background baked in rather than a transparent one, which didn't render cleanly against the new dark header.
- **Reduced left/right padding from 40px to 24px** in every header, body, and footer cell across all five templates, giving the content more usable width within the same 600px email card.

## [0.7.0] — 2026-07-05

### Added
- **Genuine sound and video submission support.** Previously "supported" only in name — the intake `Email\Parser` MIME allowlist blocked all audio/video before anything else could run, and even the description code that did exist would have discarded the actual video file. Now: `Parser::ALLOWED_MIME` accepts 8 audio and 6 video MIME types with type-appropriate size caps (20MB image / 50MB audio / 150MB video); `AI\MediaAdapter::adapt_video()` extracts a poster frame via ffmpeg when available (never drops the original video if it isn't); `AI\Pipeline::process_video_single()` describes the poster frame with the same vision call used for images, falling back to a text-only description from the artist's message when no frame is available; `Publishing\PostCreator` uploads the real audio/video binary through the same Media Library path as images and publishes it with native `core/audio`/`core/video` blocks (zero theme support required), using a video's poster frame as the post's featured thumbnail. Quality-rejection and AI enhancement never apply to audio/video — both are always published as submitted, described only, never altered.
- **Artist Guide: new "Sound and video" section** explaining that audio/video attachments are sent the same way as images, how AI describes them, and that the file itself is always published untouched.
- **External links become `wp:embed` blocks.** When an artist's message contains a link — typically a video too large to email — to an allowlisted platform (YouTube, Vimeo, Dailymotion, SoundCloud, Bandcamp, Archive.org), `PostCreator::build_external_link_embeds()` appends it as a native embed block at the bottom of the post. Deliberately allowlist-only, not a content classifier: this pipeline has no way to inspect what a linked page actually shows, so "reject commercial/adult sites" is achieved structurally by only ever trusting a short, maintained list of known platforms rather than trying to detect what to exclude. Extensible via the `agnosis_embed_host_allowlist` filter. Capped at 3 embeds per submission.

Corrected to say so explicitly.

## [0.6.3] — 2026-07-05

### Added
- **`agnosis/newsletter-popover` block — the subscribe trigger + popover chrome, now plugin-owned.** Previously this lived as theme markup (`agnosis-theme`'s `wp:html` trigger button, an anchored `wp:group` panel, and a `render_block_core/group` filter that added `popover="auto"` by regex). Moved here as `Agnosis\Newsletter\PopoverBlock`: a single self-contained dynamic block renders the trigger button, the native-Popover panel, and an `agnosis/newsletter-signup` form inside it (via `render_block()`, same as any other block would), and writes `popover="auto"` directly rather than patching it in after the fact. Lets an operator pick something that won't visually blend into whatever sits next to it (e.g. Social Share icons) without editing CSS or the theme.

## [0.6.2] — 2026-07-05

### Changed
- **"Log in to view your submissions" form restyled and gained optional Turnstile protection.** `SubmissionsPage::render_login_form()` previously emitted `wp_login_form()` completely unstyled — labels and inputs sat inline, undersized. Added scoped CSS: username/password stack label-above-input on their own lines, inputs are full-width with larger padding/font-size, matching the Join/Subscribe forms' look.

### Added

## [0.4.3] — 2026-07-04

### Changed
- **Artist subdomains no longer replace the visible site title with the artist's name.** `SubdomainRouter` filtered `option_blogname` since 0.1.9 so `wp:site-title` showed the artist rather than "Agnosis" — in practice this left the header with no visible Agnosis branding at all and, combined with `wp:site-title`/`wp:site-logo` both linking to `home_url()` (which `rewrite_home()` already points at the artist subdomain), no way to navigate back to the main portal from an artist's page. The `option_blogname` filter is removed; `document_title_parts`/`wp_title` (the HTML `<title>`, used for the browser tab/SEO) still show the artist's name and are unaffected. The actual "no way back" problem is fixed on the theme side: agnosis-theme's `functions.php` now points the Site Logo and Site Title links at the main Agnosis domain (`agnosis_base_domain` option) instead of `home_url()` whenever `SubdomainRouter::is_artist_subdomain()` is true.

### Fixed
- **Email action links executed on a plain GET — corporate mail-security scanners could trigger them.** Every one-click link in the platform (`ReviewConfirm` review approve/reject/remove, `VouchConfirm` admission votes, `SubscriptionConfirm` newsletter confirm/unsubscribe for both the public list and artists) mutated state the instant its URL was fetched, at `template_redirect`. Link-scanning mail security layers (Outlook SafeLinks/ATP, Mimecast, Proofpoint, and similar) prefetch GET links in incoming email to scan them — which is indistinguishable, server-side, from the recipient actually clicking. A prefetch alone could consume a single-use review token (the artist's real click then shows "link expired or already used"), silently approve/reject/remove artwork, cast a community admission vote on the artist's behalf, or confirm/unsubscribe a newsletter recipient who never opened the email client. Fixed by splitting every one of these shims into two phases: a GET now only renders a themed confirm page naming the action, with a single POST button; the token travels in that form as a hidden field (never in the form's `action` URL, so it is not logged a second time), and the action itself — REST dispatch, vote recording, or subscriber status change — only runs once that POST arrives. Scanners issue GETs and never submit forms, so prefetching a link is now inert. `VouchConfirm` and `SubscriptionConfirm` also had their response pages moved from raw `echo`+`exit` onto `wp_die()` (matching `ReviewConfirm`'s existing pattern), which incidentally closes a testing gap noted in both suites and lets the whole GET/POST split be verified end-to-end; `VouchConfirm`'s error page also now correctly returns HTTP 400 (previously always 200), consistent with the other two shims.
- **The public newsletter signup block never sent the visitor's language, so per-locale rendering never applied to public subscribers.** `Subscription::register_routes()` has accepted a `language` param since 0.4.0 and correctly maps it to a WP locale via `Admission::iso_to_wp_locale()`, but the signup block's `frontend.js` only ever sent `{ email }` — there was no language field, and no other source for the value. Every public-list subscriber therefore had `locale = NULL`, silently resolving to the site default locale at send time and leaving 0.4.1's per-locale digest/intro rendering exercised only by the artist audience in practice. Fixed by having `frontend.js` send `document.documentElement.lang` (the primary subtag, lower-cased — `"es-ES"` becomes `"es"`, the same split-on-hyphen normalization `Admission::apply()` already applies to its `Accept-Language` fallback) as `language` in the subscribe payload. WordPress and Lingua Forge already set `<html lang="…">` per language version of a page, so a visitor subscribing from `/es/` or an `es` subdomain now gets the right locale for free, with no new form field and no server-side change.
- **The community removal-vote email link was completely dead — no handler was ever registered for it.** `DepartureNotification::removal_vote_url()` has generated `?agnosis_removal_vote=1&rid=…&vid=…&vote=…&token=…` links in removal-vote notification emails since the removal-vote feature shipped, but no `template_redirect` handler ever consumed that query string, and there was no other UI to cast a removal vote at all — every link 404'd (or rendered the homepage), silently disenfranchising the entire community on every removal vote. Fixed by adding `RemovalVoteConfirm`, a new email-link shim mirroring `VouchConfirm`'s stateless-HMAC-token pattern and its GET/POST split from day one (a GET renders a themed confirm page; the vote is only recorded on POST). `Departure::cast_vote()` was refactored to delegate to a new shared `record_vote_on_request()` method so the REST endpoint and the email-link shim share one validation path, the same relationship `Admission::record_vote()` already has with `VouchConfirm`.
- **A failed newsletter send was given up on after a single `wp_mail()` failure, with no retry.** `QueueProcessor::process()` marked a queue row `failed` the instant `wp_mail()` returned `false` — a transient SMTP hiccup or a momentary host outbound-rate-limit rejection permanently dropped that recipient from the issue, with no second chance and no visible distinction from a genuinely bad address. Fixed with a bounded retry: `agnosis_newsletter_queue` gains an `attempts` column, and a failed send now stays `pending` (so the next 5-minute cron tick retries it) until three attempts have failed, at which point it is marked terminally `failed` and `resolved_at` records the final-failure time. Existing installs get the column via a guarded `ALTER TABLE` in `Activator::maybe_upgrade()`.
- **No RFC 8058 one-click unsubscribe.** Newsletter mail set `List-Unsubscribe:` but not the paired `List-Unsubscribe-Post: List-Unsubscribe=One-Click` header, and the unsubscribe URL only handled GET — falling short of the RFC 8058 one-click flow that Gmail/Yahoo bulk-sender rules require above ~5k messages/day and that also feeds Gmail/Outlook's built-in "Unsubscribe" affordance and spam-folder placement at any scale. Fixed by adding the `List-Unsubscribe-Post` header to every send, and giving `SubscriptionConfirm` a third path alongside its existing GET-confirm/POST-act split (§2a): a bare POST whose body is the RFC 8058-mandated literal `List-Unsubscribe=One-Click` is recognized (it can only ever come from a mail client's own automated request, never from prefetch-vulnerable GET scanning) and acted on immediately, with no confirm page — reading action/type/token/uid from the URL's query string, since that's where a header-triggered request puts them rather than in the POST body. The *visible* body unsubscribe link a human clicks is untouched and still gets the §2a confirm-button page.
- **The newsletter subscribe endpoint let anyone probe whether an email address was on the list.** `Subscriber::subscribe()` returned a distinguishable `WP_Error` (HTTP 409) for an already-confirmed address versus a 201 for anything else, so a caller could enumerate subscribers by status code alone. Fixed by making every outcome — new, still-pending, or already-confirmed — return the same success shape and the same 201 response; only whether a confirmation email goes out differs (skipped entirely when already confirmed, since there's nothing left to confirm), and that's invisible to the caller.
- **Pending subscribers and their tokens never expired, and every resubmission resent a confirmation email.** A bot hitting the signup form (still bounded by the existing 5/IP/5min rate limit) could accumulate `pending` rows forever, each triggering its own confirmation email, with no cleanup. Fixed with two independent pieces of hygiene: (1) `Subscriber::expire_stale_pending()` deletes abandoned `pending` rows older than 14 days — piggybacked on the existing daily `agnosis_prepare_newsletters` cron (`Scheduler::prepare()`) rather than a new scheduled event — and never touches `confirmed`/`unsubscribed` rows; (2) resubscribing a still-pending address within a 5-minute cooldown now leaves the row and its token completely untouched (no fresh token, no repeat email) instead of resending every time, while a resubmission past the cooldown still resends normally with a fresh token. Both the age check and the cooldown check are done in SQL against MySQL's own `NOW()` rather than PHP's clock, so there's no timezone-mixing risk of the kind fixed separately below (§3e).
- **Newsletter scheduling mixed three different clocks.** `Scheduler::since_window()`'s never-sent-before fallback used `gmdate()` (true UTC) while every other path returns `sent_at` (a WP site-local wall-clock string, from `current_time('mysql')`) — so on a site with a non-zero UTC offset, the very first digest's content window was shifted against `post_date` (also site-local), and `is_due()`'s bare `strtotime($last_sent)` silently substituted PHP's own ini `date.timezone` for the site's actual configured WordPress timezone when converting back to a Unix epoch, double-converting the offset against `time()`. Fixed by keeping every scheduling timestamp on the one clock the rest of the pipeline (`post_date`, `resolved_at`) already uses — site-local — and using WP core's own `get_gmt_from_date()` (DST-correct, via `wp_timezone()`) for the one place an actual Unix-epoch comparison is needed (`is_due()`), instead of a bare `strtotime()`.
- **Four small hygiene nits.** The newsletter signup block's email `<label>` had no `for`/`id` association with its input, so screen readers had no programmatic link between them — fixed with a matching `id`/`for` pair. `agnosis_newsletter_queue`'s `sent_at` column was renamed to `resolved_at` — it was stamped on a terminal failure just as often as a real send, and the old name was misleading; existing installs get the rename via a guarded `ALTER TABLE`. `agnosis_newsletter_subscribers.email` was shortened from `VARCHAR(255)` to `VARCHAR(191)` — its `UNIQUE` index under `utf8mb4` exceeded the 767-byte limit on older MariaDB/MyISAM configurations; 191 (WordPress core's own long-standing convention for indexed email columns) stays safely under it everywhere. The `SubscriptionConfirm::render_page()` hardcoded-`lang="en"` nit turned out to already be resolved as an incidental side effect of the §2a `wp_die()` conversion earlier in this release — no further change was needed there.

## [0.4.2] — 2026-07-02

### Fixed
- **A newsletter issue could get stuck in `sending` forever when every recipient's send failed.** `QueueProcessor::process()` previously only re-checked (and possibly marked `sent`) issues that had at least one row transition to `sent` in that batch (`$sent_per_issue`). If every recipient's `wp_mail()` call failed — e.g. a single-artist newsletter hitting a temporary mail delivery problem — none of its rows ever reached `sent`, so the issue was never reconciled; and since its rows were already `failed` (not `pending`), no future cron tick would ever look at it again either. The admin dashboard showed that newsletter as "Sending…" indefinitely, with "Send Now" disabled (`has_issue_in_flight()` blocks a second send while one is "in flight"). Fixed by having `process()` reconcile **every** currently-`sending` issue each tick via a new `reconcile_sending_issues()` pass, regardless of whether that batch touched it or whether there's anything left to send — self-healing any issue already stuck this way, not just preventing new ones.

## [0.3.0] — 2026-06-30

### Changed

Publishing an Agnosis post no longer makes AI calls inline. `agnosis_post_published` now only schedules a single WP-Cron event (`agnosis_dispatch_lf_translations`); the actual work — building per-language artwork titles and requesting the body translations — runs later in `dispatch_translations()`. The body translations now use LF 2.4.0's async `linguaforge_queue_translation()` (Action Scheduler / WP-Cron) when available, falling back to the synchronous trigger on older LF. Together this takes a webhook/IMAP intake from ~2N blocking AI calls down to zero on the intake request. (audit §3f)

Events use LF's normal title translation (an event title is translatable content); biographies likewise — the artist's name is set at application time and is not derived from the biography post title, so it needs no special protection. The §2a image/meta propagation stays scoped to all three CPTs. (audit §3f)

### Added
- **Two out-of-the-box content pages.** On activation Agnosis now creates an **About** page (`/about/`) explaining what the community is and how it works, and an **Artist Guide** help page (`/artist-guide/`) walking an artist through the email-driven workflow (submitting artwork, the photo-only lane, biographies, events, updating/removing, writing in your own language). Both are managed pages (idempotent on re-activation, tagged `_agnosis_managed_page` so `uninstall.php` removes them) authored in plain language; Lingua Forge translates them like any page, so no separate i18n is needed for the copy. (audit §9)
- **Member-governed community size cap.** Each instance has a maximum number of admitted artists (`agnosis_community_max_artists`, default 50; 0 = unlimited), and the community — not a hard-coded number — controls it.
  - *Phase 1 (cap + waitlist):* a new `CommunityCap` gate. When the instance is full, applications join a FIFO **waitlist** (`waitlisted` status) instead of being rejected; the cap gates new admissions only, so existing members are always grandfathered and lowering the cap never removes anyone. The gate is enforced at `apply()` and re-checked at `maybe_admit()` (a pending application that reaches its vouch threshold while full is parked, not admitted). When a member permanently leaves (`agnosis_artist_left` / `agnosis_removal_vote_passed` / `agnosis_artist_deleted_by_admin`), the oldest waitlisted applicant is advanced back into the vouching flow ("open a slot, fill a slot") and admitted at once if it already had enough vouches. Admins may admit over the cap (logged). Configurable in Settings → Network; the join form reports the full/waitlist state.
  - *Phase 2 (community vote):* a new `CommunityCapVote` governance flow mirroring the removal-vote machinery — any artist proposes a new cap (`POST /agnosis/v1/cap/propose`), it opens as a full community vote once it gathers the configured co-signers (`agnosis_cap_proposal_threshold`, default 3), members vote (`/cap/cosign/{id}`, `/cap/vote/{id}`), and the daily `agnosis_check_cap_votes` cron closes the window: a strict majority (> 50 %) of active artists voting yes writes the new cap and advances the waitlist. New tables `agnosis_cap_proposals` + `agnosis_cap_votes`; `CommunityCapNotification` emails members when a vote opens and the admin on resolution. (audit §8) *(A member-facing voting block + one-click email links — mirroring the removal-vote HMAC links — are a noted follow-up; the governance backend and REST API are complete.)*
- **`uninstall.php` — complete teardown on delete.** Deleting the plugin now removes its full infrastructure footprint: the 9 custom tables, every `agnosis_*` option and transient, the `agnosis_artist` role, the managed pages (`/join/`, My Submissions), all scheduled cron events, and the `uploads/agnosis-queue/` working directory. Multisite-aware (runs per site). Deliberately preserves the artists'/operator's content — published artwork/biography/event posts and their media, and WordPress user accounts (removing the role drops only its capabilities). Previously `deactivate()` cleared cron only, so everything else was orphaned on delete. (audit §C-2)
- **Per-language artwork titles across translations** (`Compat/LinguaForge.php`). The dual-title architecture (`post_title` = artist's own-language title; primary/translated title in `_agnosis_translated_title`) now carries correctly to LF translations. (1) `hold_artist_title()` hooks `linguaforge_translation_content` to keep `post_title` as the artist's original on every language version — previously LF translated that non-primary title from the wrong source language. (2) `build_title_translations()` runs at publish (`agnosis_post_published` priority 15) and translates the primary-language title into each enabled LF language via `SubmissionTranslator`, storing a `_agnosis_title_i18n` map. (3) `supply_translated_meta()` / `copy_translated_meta()` write each translation's `_agnosis_translated_title` from that map. Solved entirely with existing LF hooks (`linguaforge_translation_content` + the 2.4.0 `linguaforge_translated_post_meta`) — no new LF API. Graceful no-op when no AI provider is configured. (audit §2d title note)
- **Translated-post image/meta propagation** (`Compat/LinguaForge.php`). Translated `agnosis_artwork` / `agnosis_biography` / `agnosis_event` posts created by Lingua Forge now inherit the source post's images and other language-neutral meta, so they no longer render empty. A curated allowlist (`NEUTRAL_META_KEYS`: `_thumbnail_id`, `_agnosis_gallery_ids`, `_agnosis_original_title`, `_agnosis_event_location`, `_agnosis_event_date`) is copied — deliberately **not** "all `_agnosis_*`", which would leak per-post security tokens (`_agnosis_review_token` / `_agnosis_removal_token`), corrupt duplicate-detection identity (`_agnosis_queue_id`, `_agnosis_image_hash`), and carry language-specific values (`_agnosis_translated_title`, `_agnosis_detected_lang`) onto the wrong-language post. On LF 2.4.0+ the allowlist is supplied via the `linguaforge_translated_post_meta` filter so the post is born complete (no empty-image window); on older LF it falls back to the `linguaforge_translation_complete` action (which also refreshes on re-translation). Image alt text is not copied — translations reuse the same attachment IDs, so it travels with the attachment. (audit §2a)

## [0.1.8] — 2026-06-28

### Security
- **Token generation** — review and removal tokens now use `bin2hex(random_bytes(32))` (256-bit CSPRNG) instead of `wp_hash()` (HMAC-MD5 over predictable inputs). Tokens cannot be reconstructed from artist ID, timestamp, or site secrets. Extracted into `PostCreator::generate_token()` and covered by `TokenGenerationTest` (format, uniqueness, `hash_equals` compatibility).
- **REST endpoint permission gates** — `ReviewEndpoints` (`PUT /review/{id}`, `POST /review/{id}/approve`, `POST /review/{id}/reject`) and `RemovalEndpoints` (`POST /removal/{id}/confirm`) previously used `'permission_callback' => '__return_true'`, relying entirely on the internal `check_access()` / `verify_token()` logic. Added `check_permission()` to each class as an explicit outer gate: returns `true` when a token param is present or the user is logged in, `WP_Error(401)` otherwise.
- **Token URL shim** — email action links (approve, reject, remove) previously linked directly to REST endpoints with `?token=` in the query string; tokens appeared in server access logs, browser history, and Referer headers. Added `ReviewConfirm` class, registered on `template_redirect`: email links now point to `/?agnosis_review=1&id={id}&action={action}&token={token}`. The shim calls `rest_do_request()` with the token in the POST body (no logged REST URL) and immediately redirects to `/?agnosis_result={action}` — a clean URL. Browser history and Referer headers never contain the token.
- **ActivityPub inbox HTTP Signature verification** — `POST /activitypub/inbox` previously accepted any payload from any IP. Added `HttpSignature` class implementing Fediverse-compatible HTTP Signature verification: parses `keyId`/`headers`/`signature` from the `Signature` header; enforces a ±12 h Date freshness window; verifies the `Digest: SHA-256=…` body hash when present; fetches `publicKey.publicKeyPem` from the sender's actor document (cached 24 h per keyId); reconstructs the signing string and verifies RSA-SHA256 via `openssl_verify()`. Wired as `verify_inbox_signature()` permission callback on the inbox route — a spoofed or unsigned Follow/Announce is rejected with 401/403 before `inbox()` runs.

### Security (continued)
- **Admission status endpoint now requires authentication** — `GET /agnosis/v1/admission/status/{id}` previously returned `is_artist`, `has_applied`, and vouch counts for any WP user ID without authentication, allowing enumeration of the entire artist/applicant roster. Added `require_logged_in_for_status()` permission callback (returns 401 for anonymous requests) and an ownership guard inside `status()`: a non-admin user requesting a different user's status receives 403. Administrators retain full access. Covered by `AdmissionStatusTest` (7 assertions: 401 anonymous, 403 cross-user, 200 own data, field correctness, admin access to any user, admin field accuracy, artist own status).
- **Per-IP rate limiting on webhook and REST endpoints** — new `Core\RateLimiter` class uses WordPress transients to count requests per action per IP within a rolling time window. The window slot rotates automatically via `floor(time() / $window)` so counters expire without a cleanup cron. Limits: `POST /admission/apply` → 5 req/min (wired in `Admission::require_logged_in()`); `POST /admission/vouch/{id}` → 20 req/min (in `Admission::require_artist()`); `POST /email/inbound` → 60 req/min (in `Webhook::verify_signature()`). - **Per-row queue actions** — Process (pending/failed rows), Reprocess (published rows), Delete, and View Post buttons on each inbox row
- **`chat()` method on AI providers** — lightweight text-only completion (OpenAI `gpt-4o-mini` / Anthropic `claude-haiku-4-5-20251001` / WordPress AI Client) for tasks that don't require vision
- **AI-powered duplicate detection** (`PostCreator::find_duplicate_post()`) — after the pipeline runs, compares new submission title + tags against recent same-artist artwork posts using the cheap text model; merges into the existing post if a match is found
- **Image hash duplicate detection** — MD5 of each attachment's raw binary is stored on the attachment (`_agnosis_image_hash`) and mirrored to the artwork post; checked before the AI call for exact-match resends (most reliable signal, zero cost)
- **Series amplification workflow** — when an artist resubmits a series with old images + new ones, existing attachments are reused (no re-upload), and new images are appended to the existing gallery
- **Subject-line indicators** — `[Biography]` and `[Event]` prefixes in the email subject route submissions to dedicated CPTs; indicator is stripped before the clean subject is used as the post title; unknown indicators fall back to `agnosis_artwork`
- **`agnosis_biography` CPT** — singleton per artist; always updates the single existing biography post; no archive
- **`agnosis_event` CPT** — singleton per artist; always updates the single existing events page; no archive
- **`Pipeline::polish()`** — lightweight AI text pass (spelling + grammar only, no rewrites); used by singleton submissions when polishing is enabled
- **AI polish settings** — two checkboxes in the AI tab ("Polish biography with AI", "Polish events with AI"); off by default; when enabled the email body is polished before saving

### Changed
- **Status badge semantics** — queue status `published` now reflects actual WP post status: "Awaiting Review" (draft), "Live" (publish), or "Processed" (other); was always showing "Published" regardless of post state
- **Admin JS** — rewrote AI provider test from jQuery `$.post()` to vanilla `fetch()` + `URLSearchParams`; script dependency changed from `jquery` to `wp-util`
- **Post idempotency** — `PostCreator::create_post()` queries `_agnosis_queue_id` meta before inserting; reprocessing a queue row updates the existing post in-place instead of creating a duplicate

### Fixed
- `is_already_queued()` SELECT failed on tables missing the `post_id` column (added in 0.1.3), causing `get_row()` to return null and every message to re-enqueue
- `enqueue()` INSERT silently no-oped when `is_already_queued()` false-returned for existing rows; replaced with `INSERT IGNORE` + follow-up SELECT for true idempotency
- Binary attachment data corrupted by `wp_json_encode()` — function runs UTF-8 sanitisation which destroys JPEG/PNG bytes; attachments are now base64-encoded before JSON storage and decoded in `PostCreator::handle()`
- Empty artwork posts (title only, no body/tags/images) caused by the above corruption silently failing the AI pipeline
- `wp_tempnam()` / `wp_handle_sideload()` undefined in cron and `admin-post` contexts — added `require_once ABSPATH . 'wp-admin/includes/file.php'` before first use
- Settings page asset enqueue hook corrected to `agnosis_page_agnosis-settings` after submenu restructure (was `toplevel_page_agnosis-settings`)
- `wp_get_post_tags()` return typed as `array|WP_Error`; guarded with `is_array()` before passing to `implode()`
