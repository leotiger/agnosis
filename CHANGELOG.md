## [0.1.9] — 2026-06-28

### Added
- **`/my-submissions` works on artist subdomains** — `SubdomainRouter::scope_query()` previously applied the `author` filter to every main query, including static page requests. Since the `/my-submissions` page is created by the system (not the artist user) it returned a 404 on any artist subdomain. A guard now skips the author and post_type filters when `pagename` is set or the query already targets `post_type=page`, letting WordPress resolve sitewide pages normally. Session cookies must also carry across subdomains — add `define('COOKIE_DOMAIN', '.agnosis.art');` to `wp-config.php` so artists logged in on the main domain are also authenticated on their subdomain.
- **Inline login form on `/my-submissions`** — unauthenticated visitors no longer see a plain "Please log in" notice or get bounced to `wp-login.php`. `SubmissionsPage` now renders an inline form via `wp_login_form()` with the current page URL as the redirect, so the artist logs in and lands back on their submissions page without ever leaving the frontend.
- **Artist name in subdomain site title** — `SubdomainRouter` now filters `option_blogname` on artist subdomains so the `wp:site-title` block (and anything else reading `get_bloginfo('name')`) displays the artist's display name instead of the main portal site name. Falls back to the nicename slug so the header is never blank. The existing `document_title_parts` and `wp_title` filters (HTML `<title>`) already did this; `option_blogname` closes the gap for the visible header.
- **Multilingual submission translation** — artist emails are automatically translated to the site's primary language before AI processing. The target language is resolved from the Lingua Forge `linguaforge_primary_language` option, with fallback to the WordPress site locale and a final fallback to English. Translation is performed in a single `chat()` call (one round-trip) using a JSON envelope that carries both the email subject and body. If translation fails (empty or non-JSON response) the original text is passed through unchanged — the pipeline always continues. The target language is extensible via the `agnosis_translation_languages` filter. Translation events are logged to Settings → Logs (`pipeline` context). Implemented in new `SubmissionTranslator` class (`includes/AI/SubmissionTranslator.php`); `chat()` max_tokens bumped from 512 → 1024 in OpenAI and Anthropic to accommodate translated email bodies.
- **Media type expansion** — the AI pipeline now handles audio, video, and PDF attachments in addition to images:
  - **PDF portfolios** — each page is rasterised to a JPEG at 150 dpi using the PHP Imagick extension and fed into the normal describe → optionally-enhance image branch. Multi-page PDFs produce one gallery image per page. Falls back gracefully (skips with a logged notice) when Imagick is not installed.
  - **Video stills** — the first frame is extracted via `ffmpeg` and processed as a JPEG through the image branch. Enhancement is applied normally. Falls back gracefully when `ffmpeg` is not in `PATH`.
  - **Audio files** — routed through a new audio branch: (1) transcribed via OpenAI Whisper (`POST /v1/audio/transcriptions`, `whisper-1`); (2) the transcript + artist context is sent to the text model (`chat()`) with a structured JSON prompt to produce `title`, `excerpt`, `body`, `tags`, `alt_text`, and `medium`; (3) no image is uploaded to the Media Library — `PostCreator::merge_gallery()` skips `upload_image()` for `media_type === 'audio'` results. Providers without audio support (`Anthropic`, `WordPressAI`) return an empty transcript and fall back to describing from artist text alone.
  - **`MediaAdapter`** — new `includes/AI/MediaAdapter.php` class with a single static `adapt(array $attachments): array` method. Sits in front of the pipeline loop; expands each attachment into zero or more normalised entries with a `media_type` field (`'image'` or `'audio'`). Unsupported MIME types are logged and dropped.
  - **`ProviderInterface`** extended with `transcribe(string $audio_data, string $mime_type): string` and `supports_audio(): bool`; existing providers updated.
- **Automatic quality rejection gate** — when the primary image's quality score is at or below the configurable rejection threshold (default: 3 / 10), the submission is rejected automatically: the queue row is marked `failed`, no draft post is created, and the artist receives a friendly HTML email explaining the specific photographic issue and how to fix it. Issue labels from the vision AI (e.g. "too dark", "motion blur", "glare") are mapped to plain-language advice with concrete tips ("Try photographing near a large window in daylight…"). The email also includes four universal photography tips. Artists are invited to resubmit — the entire flow is automated with no admin action required. Configurable via Settings → AI Providers: **Rejection threshold** (set to 0 to disable). Covered by `QualityRejectionTest` (5 assertions: score at threshold rejects, score below threshold rejects, score above threshold proceeds, score 0 skips gate, threshold 0 disables gate entirely). — when the primary image's quality score is at or below the configurable rejection threshold (default: 3 / 10), the submission is rejected automatically: the queue row is marked `failed`, no draft post is created, and the artist receives a friendly HTML email explaining the specific photographic issue and how to fix it. - **`Pipeline::extract_event_fields()`** — lightweight chat pass that runs after the polish step for `agnosis_event` submissions; sends body + subject to the text model and returns a JSON object with a `location` key; strips markdown fences from the response; short-circuits (returns `['location' => '']`) when body and subject are both blank. No admin entry required — location is fully automated.
- **`_agnosis_event_location` post meta** — stored on every `agnosis_event` post via `meta_input` during `PostCreator::create_post()`; populated from `Pipeline::extract_event_fields()` and falls back to an empty string when the AI finds no location in the email.
- **`agnosis/event-location` block** — server-side-rendered block registered in `Profile::register_blocks()`; uses `postId` context from the query loop; reads `_agnosis_event_location` meta and renders a styled `<p class="agnosis-event-location">` (small, bold, uppercase, letter-spaced); returns an empty string when meta is unset so events without a location leave no visual gap.
- **`agnosis_email_promote` setting** (Email tab) — configurable promote@ address (e.g. `promote@agnosis.art`); description explains the workflow: artist sends an email with the exact artwork title as subject to explicitly feature that artwork in their gallery overview.
- **`PostCreator::handle_promotion_request()`** — handles the `agnosis_promote` pseudo-type; finds the sending artist's published artwork by exact title match; calls `set_featured()` if found; logs a warning and exits silently when subject is empty or no match is found.
- **`PostCreator::set_featured()`** — sets `_agnosis_featured = '1'` on the target artwork and demotes all other published artworks by the same artist to `'0'`; drafts and other artists are not affected. Kept in the publishing layer (not `GalleryOverview`, which is a display class) because featuring is a workflow decision.

### Fixed
- **`CREATE TABLE IF NOT EXISTS` removed from `dbDelta()` calls** — `dbDelta()` handles the "table already exists" case itself by diffing the schema against the current table; the `IF NOT EXISTS` clause caused it to short-circuit before performing that comparison, silently skipping new columns on existing tables. All five `CREATE TABLE IF NOT EXISTS` statements in `Activator::create_tables()` are now plain `CREATE TABLE`. The pre-existing manual `ALTER TABLE` guards in `maybe_upgrade()` (`post_id`, `revoked_at`) are retained for backward compatibility with installs created before this fix.

### Changed
- **Auto-promotion on approval removed** — `ReviewEndpoints::approve()` and `save(publish=true)` no longer set `_agnosis_featured`; the entire `promote_featured()` method has been deleted. Featuring is now always explicit — the artist sends an email to the promote@ alias with the artwork title as subject.
- **Theme 0.1.5** — `single-agnosis_biography.html` rebuilt with a two-column layout (left 38%: featured image at 3/4 aspect, 4 px border-radius; right 62%: title + spacer + content; columns stack on mobile; separator + author/year footer preserved); `single-agnosis_event.html` updated to group the post date and `wp:agnosis/event-location` block together with reduced block gap.

### Fixed
- **`ReviewEndpoints::reject()` incorrect action arg count** — `do_action('agnosis_submission_rejected', $post_id)` was passing only 1 argument; hooks registered with `accepted_args=4` received `$post_id` in the `$score` parameter position instead of `$author_id`. Fixed to pass all four expected args: `$post_id, (int) $post->post_author, 0, []`.
- **`render.php` unescaped `$agnosis_thumb_id`** — attachment ID in `class="wp-image-{id}"` was emitted raw; wrapped with `absint()`.
- **Minimum WordPress version bumped to 6.6** — `wp_interactivity_state()` and `wp_enqueue_script_module()` were both introduced in WP 6.5; declaring 6.4 as minimum caused Plugin Check errors. Updated plugin header, `AGNOSIS_MIN_WP` constant, and `readme.txt`.

## [0.1.8] — 2026-06-28

### Security
- **Token generation** — review and removal tokens now use `bin2hex(random_bytes(32))` (256-bit CSPRNG) instead of `wp_hash()` (HMAC-MD5 over predictable inputs). Tokens cannot be reconstructed from artist ID, timestamp, or site secrets. Extracted into `PostCreator::generate_token()` and covered by `TokenGenerationTest` (format, uniqueness, `hash_equals` compatibility).
- **REST endpoint permission gates** — `ReviewEndpoints` (`PUT /review/{id}`, `POST /review/{id}/approve`, `POST /review/{id}/reject`) and `RemovalEndpoints` (`POST /removal/{id}/confirm`) previously used `'permission_callback' => '__return_true'`, relying entirely on the internal `check_access()` / `verify_token()` logic. Added `check_permission()` to each class as an explicit outer gate: returns `true` when a token param is present or the user is logged in, `WP_Error(401)` otherwise.
- **Token URL shim** — email action links (approve, reject, remove) previously linked directly to REST endpoints with `?token=` in the query string; tokens appeared in server access logs, browser history, and Referer headers. Added `ReviewConfirm` class, registered on `template_redirect`: email links now point to `/?agnosis_review=1&id={id}&action={action}&token={token}`. The shim calls `rest_do_request()` with the token in the POST body (no logged REST URL) and immediately redirects to `/?agnosis_result={action}` — a clean URL. Browser history and Referer headers never contain the token.
- **ActivityPub inbox HTTP Signature verification** — `POST /activitypub/inbox` previously accepted any payload from any IP. Added `HttpSignature` class implementing Fediverse-compatible HTTP Signature verification: parses `keyId`/`headers`/`signature` from the `Signature` header; enforces a ±12 h Date freshness window; verifies the `Digest: SHA-256=…` body hash when present; fetches `publicKey.publicKeyPem` from the sender's actor document (cached 24 h per keyId); reconstructs the signing string and verifies RSA-SHA256 via `openssl_verify()`. Wired as `verify_inbox_signature()` permission callback on the inbox route — a spoofed or unsigned Follow/Announce is rejected with 401/403 before `inbox()` runs.

### Security (continued)
- **Admission status endpoint now requires authentication** — `GET /agnosis/v1/admission/status/{id}` previously returned `is_artist`, `has_applied`, and vouch counts for any WP user ID without authentication, allowing enumeration of the entire artist/applicant roster. Added `require_logged_in_for_status()` permission callback (returns 401 for anonymous requests) and an ownership guard inside `status()`: a non-admin user requesting a different user's status receives 403. Administrators retain full access. Covered by `AdmissionStatusTest` (7 assertions: 401 anonymous, 403 cross-user, 200 own data, field correctness, admin access to any user, admin field accuracy, artist own status).
- **Per-IP rate limiting on webhook and REST endpoints** — new `Core\RateLimiter` class uses WordPress transients to count requests per action per IP within a rolling time window. The window slot rotates automatically via `floor(time() / $window)` so counters expire without a cleanup cron. Limits: `POST /admission/apply` → 5 req/min (wired in `Admission::require_logged_in()`); `POST /admission/vouch/{id}` → 20 req/min (in `Admission::require_artist()`); `POST /email/inbound` → 60 req/min (in `Webhook::verify_signature()`). Exceeding the limit returns `WP_Error('agnosis_rate_limit', …, ['status' => 429])`. IP detection uses `REMOTE_ADDR` only — not `X-Forwarded-For` — to prevent header spoofing; sites behind a trusted reverse proxy that rewrites `REMOTE_ADDR` (nginx `real_ip_from`, AWS ALB, Cloudflare) work correctly without configuration.

### Changed
- **`WP_Filesystem` used for all filesystem writes** — replaced bare `file_put_contents()` / `file_get_contents()` calls in `PostCreator::upload_image()` and `Email\AttachmentStore` with `WP_Filesystem` (`'direct'` method forced via filter, since FTP/SSH credentials cannot be prompted in cron or admin-post context). Extracted as a private `filesystem()` helper on `PostCreator` and an equivalent static helper on `AttachmentStore`.
- **Attachment binaries moved out of the database** — `Inbox::enqueue()` now writes each image binary to `uploads/agnosis-queue/{uid}/{index}-{filename}` via the new `Email\AttachmentStore` class; `raw_email` JSON stores a `file` path instead of a base64 blob, eliminating 5–10 MB DB payloads per submission. `PostCreator` reads the file and deletes it after a successful media library upload. `AttachmentStore::sweep_orphans()` is called by the daily cleanup cron to remove dirs left by failed rows. The queue directory is protected with `.htaccess` and `index.php` on activation. Rows enqueued before this change (base64 inline) continue to work via a backward-compat branch.
- **Vouch revocation** — added `revoked_at DATETIME NULL DEFAULT NULL` to `agnosis_vouches`; existing installations receive the column via `maybe_upgrade()` on the next page load. `count_vouches()` filters `AND revoked_at IS NULL`. New `Admission::revoke_vouch(int $voucher_id, int $candidate_id): bool` sets `revoked_at` on the matching active row (row preserved for audit history; returns false when not found or already revoked). Covered by `VoucheRevocationTest`.
- **`handle_test_ai()` duplicate HTTP logic extracted** — the identical `wp_remote_post` + error-handling block that was copy-pasted for OpenAI and Anthropic is replaced by a shared private `Settings::ping_provider()` method. Each provider case now passes its specific URL, headers, and body; the error-handling path (transport failure → 429 error message, non-200 → extract `error.message` or fall back to `HTTP {code}`) lives in one place. Adding a fourth provider now costs 5 lines, not 18.
- **Automated queue drain on every poll** — `Inbox::drain_pending()` queries all `pending` rows from `agnosis_queue` (up to 50 per run, staggered 1 s apart) and schedules a `wp_schedule_single_event('agnosis_publish_submission', [$id])` for each. Called at the end of every `poll()` run regardless of IMAP configuration. Covers webhook-sourced rows whose initial single event was missed due to no page load after ingest, and sites running in webhook-only mode. `wp_schedule_single_event` deduplicates identical events already queued so this is safe to call every 5 minutes. Covered by `QueueDrainTest`.
- **`maybe_upgrade()` uses `SHOW COLUMNS` instead of `information_schema`** — both column-existence checks (`post_id` on `agnosis_queue`, `revoked_at` on `agnosis_vouches`) previously queried `information_schema.columns`, which is restricted on managed database providers (PlanetScale, Kinsta DB, etc.). Replaced with `$wpdb->get_results("SHOW COLUMNS FROM {$table} LIKE '{$column}'")`: returns empty on absence, works without elevated DB privileges, and is compatible with all MySQL/MariaDB hosting environments.
- **`upload_size_limit` filter cleanup scoped to own closure** — `PostCreator::upload_image()` was calling `remove_all_filters('upload_size_limit')` after `wp_handle_sideload()`, silently removing any filters added by co-active plugins. Changed to assign the closure to `$size_filter` and call `remove_filter('upload_size_limit', $size_filter)` — surgical removal of only our filter.
- **`no_found_rows` on all ID-only `get_posts()` calls** — added `'no_found_rows' => true` to 8 calls across `PostCreator` (5), `Admin\InboxPage` (2), and `Network\ActivityPub` (1) that fetch IDs and never read `$query->found_posts`. Eliminates the `SELECT COUNT(*)` subquery WP runs unconditionally otherwise; measurable on large post tables.
- **`PostCreator` Pipeline dependency injection** — constructor changed from `__construct()` to `__construct(?Pipeline $pipeline = null)`. Production code passes nothing; tests inject a stub that skips provider resolution and returns canned results without hitting AI endpoints. Covered by `PipelineInjectionTest`.
- **`PostCreator::create_post()` refactored** — reduced from 157 lines to 67 lines by extracting three private helpers: `merge_gallery()` (hash-aware gallery merge and upload), `build_post_content()` (Gutenberg block assembly), and `write_post_meta()` (featured image, tags, medium taxonomy, image hash mirroring, quality meta). No behaviour changes.
- **`Plugin.php` reduced to pure orchestrator** — extracted all business logic out of the bootstrap class. New `Admin\QueueController` takes the five queue admin-post handlers (`poll_now`, `force_reprocess`, `process_queue`, `process_one`, `delete_queue_row`); `handle_test_inbox` moved to `Admin\InboxPage`; `handle_test_ai` and `handle_clear_logs` moved to `Admin\Settings`; `compatibility_notices()` moved to `Compat\LinguaForge`; `ensure_roles()` extracted to new `Core\Roles`; `register_image_sizes()` extracted to new `Core\ImageSizes`. `Plugin.php` is now 145 lines of hook wiring only.
- **Gallery overview — WP Interactivity API lightbox** — replaced the custom `<dialog>`-based lightbox with WordPress core's built-in `core/image` lightbox. Each artwork card is now a `<figure class="wp-block-image wp-lightbox-container">` with the full set of `data-wp-*` directives; `wp_interactivity_state('core/image', [...])` registers per-image metadata (full-resolution `uploadedSrc`, dimensions); `block_core_image_print_lightbox_overlay()` prints the WP overlay once via `wp_footer`; `@wordpress/block-library/image/view` script module is enqueued.
- **Gallery overview — artist label routing** — on the main domain, the artist name links to the artist's subdomain; on an artist subdomain, it links to their biography page (`/biography/{slug}/`), falling back to the subdomain home if no biography exists yet. Uses `SubdomainRouter::is_artist_subdomain()`; memoised per request.

## [0.1.7] — 2026-06-27

### Fixed

## [0.1.6] — 2026-06-27

### Added
- **Photo quality detection** — the vision AI now returns a `photo_quality` object (`score` 1–10 + `issues` array) as part of every description call; no extra API cost — the model assesses the photograph while analysing the artwork
- **Conditional image enhancement** — enhancement only runs when the photo quality score falls below a configurable threshold (default 7); good photographs are left untouched; the enhancement prompt is built from the specific detected issues ("underexposed shadows", "camera blur") rather than a generic instruction, keeping corrections targeted and the artwork itself unaltered
- **`agnosis-email` image size** — 420 px wide, proportional (no crop); used in artist notification emails; configurable via Settings → Behaviour
- **Full gallery in review email** — all submitted images (from `_agnosis_gallery_ids`) are rendered at `agnosis-email` size and stacked vertically in the notification; previously only the featured image was shown; falls back to featured image on legacy posts
- **Enhancement notice in review email** — when enhancement ran, the artist's email includes a panel listing the photographic issues that were corrected and reassuring them the artwork itself was not altered
- **Quality score post meta** — `_agnosis_photo_quality_score`, `_agnosis_photo_quality_issues`, `_agnosis_enhanced` stored on every artwork post for future admin display and reporting
- **Enhancement threshold setting** — "Enhancement threshold" field in Settings → AI Providers; score below threshold triggers enhancement; set to 1 to disable automatic enhancement
- **`agnosis-artwork` image size** — width-constrained (default 1920 px), height scales proportionally; used in post content blocks, lightbox, ActivityPub federation, and LinguaForge sharing; WP generates `srcset` automatically
- **`agnosis-thumb` image size** — 512 × 512 px square hard-crop (centred); used in submission cards and dashboard
- **Image size settings** (Behaviour tab) — "Artwork display width", "Thumbnail size", and "Email image width" let admins tune all three sizes without code changes; existing uploads need regeneration after a change
- **Dedicated email endpoints** — routing is now by `To:` header rather than subject-line indicators; five addresses configurable in Settings → Email: `submit@` (artwork), `bio@` (biography), `event@` (event), `replace@` (explicit artwork replacement), `remove@` (takedown request); subject-line indicators remain as a backward-compatible fallback
- **`replace@` endpoint** — skips AI fuzzy duplicate detection entirely; finds the existing post by exact title match across draft/pending/publish statuses and updates it in-place; intended for artists who want a deliberate replacement without ambiguity
- **`agnosis_medium` taxonomy** — 8 canonical terms seeded on activation (Oil, Acrylic, Watercolour, Drawing, Digital, Photography, Sculpture, Mixed Media); AI picks exactly one from the canonical list; term is validated against `PromptConfig::CANONICAL_MEDIUMS` before `wp_set_object_terms()` to prevent hallucinated entries creating rogue taxonomy terms
- **Medium in AI pipeline** — `DescriptionResult` carries a `medium` field; all three providers (OpenAI, Anthropic, WordPress AI) parse and return it; `PromptConfig` exposes `CANONICAL_MEDIUMS` and injects the list into the system prompt via `{medium_list}` substitution
- **Artist-driven removal workflow** — `remove@` email triggers `PostCreator::handle_removal_request()`: generates a signed `_agnosis_removal_token` (7-day expiry), stores the artist's stated reason in `_agnosis_removal_reason`, fires `do_action('agnosis_removal_requested')`; `Notification::on_removal_requested()` sends a confirmation email with a single signed "Yes, remove this artwork" button; `RemovalEndpoints` (`POST /agnosis/v1/removal/{id}/confirm?token=`) validates the token, consumes it, trashes the post, and fires `agnosis_post_removed` — no admin involvement at any step
- **Agnosis Theme** (`agnosis-theme/`) — theme folder renamed from `agnosis-feature`; `readme.txt` updated to Agnosis Theme branding, removed unused Averia Serif Libre and Teko font attributions, kept Labrada and Plus Jakarta Sans; attribution chain documented (Agnosis Theme → Feature → Issue, both Automattic)

- **Image hash duplicate detection** — MD5 of each attachment's raw binary is stored on the attachment (`_agnosis_image_hash`) and mirrored to the artwork post; checked before the AI call for exact-match resends (most reliable signal, zero cost)

- **Admin JS** — rewrote AI provider test from jQuery `$.post()` to vanilla `fetch()` + `URLSearchParams`; script dependency changed from `jquery` to `wp-util`
- **Post idempotency** — `PostCreator::create_post()` queries `_agnosis_queue_id` meta before inserting; reprocessing a queue row updates the existing post in-place instead of creating a duplicate

### Fixed
- `is_already_queued()` SELECT failed on tables missing the `post_id` column (added in 0.1.3), causing `get_row()` to return null and every message to re-enqueue
- `enqueue()` INSERT silently no-oped when `is_already_queued()` false-returned for existing rows; replaced with `INSERT IGNORE` + follow-up SELECT for true idempotency
- Binary attachment data corrupted by `wp_json_encode()` — function runs UTF-8 sanitisation which destroys JPEG/PNG bytes; attachments are now base64-encoded before JSON storage and decoded in `PostCreator::handle()`
- Empty artwork posts (title only, no body/tags/images) caused by the above corruption silently failing the AI pipeline
- `wp_tempnam()` / `wp_handle_sideload()` undefined in cron and `admin-post` contexts — added `require_once ABSPATH . 'wp-admin/includes/file.php'` before first use
- Settings page asset enqueue hook corrected to `agnosis_page_agnosis-settings` after submenu restructure (was `toplevel_page_agnosis-settings`)
- `wp_get_post_tags()` return typed as `array|WP_Error`; guarded with `is_array()` before passing to `implode()`
